# Structured Document Rules

## 用途

- **構造化文書** を lean に保つための汎用規則。
- 構造化文書の作成、更新、レビュー、リバイズで、冗長性、構造、用語・表記の厳密化、判断が必要な場合に読む。
- **構造化文書**: Markdown、LaTeX、その他の markup または source 形式で書かれ、section、list、environment、command、macro、record、schema などで意味境界を表す文書。
- **対象例**: `SKILL.md`、bundled resources、プロンプト本文、仕様書、手順書、レビュー文書、LaTeX 原稿。
- **参照例**: この file 自体も **構造化**、**冗長性の排除**、**厳密化** を適用した例として参照できる。

## 目次

- **Notation**
- **基本原則**

- **構造化**

- **内容追加、レビュー、リバイズへの適用**

## Notation

- **description item**: label と value の組で、対象、条件、例などの属性を示す item。
  - **例**: `**label**: value`
- **field**: description item または record の中で使う属性名。
- **field set**: 同じ対象群に対して使う field の集合。
- **field order**: 同じ field set の field を並べる順序。
- **unordered description**: description item を順序なしで並べたもの。
- **ordered description**: description item を順序付きで並べたもの。
- **record**: 複数対象を同じ **field set** で扱うとき、各対象を parent item とし、その child structure に `field: value` 形式の description item を並べた構造。
  - **例**:
    - **target1**:
      - **field A**: value A
      - **field B**: value B
    - **target2**:
      - **field A**: value A
      - **field B**: value B

## 基本原則

- 文書は、人間と LLM が同じ構造、用語、参照関係をたどれるように書く。
- **解釈の不確実性の減少**: 対象、目的、前提、定義、主張、根拠、結論、責務境界を明示し、同じ語を同じ意味で使う。
- **読解コストの減少**: 読む順序、階層、比較軸、例と本則の関係を文書構造で示し、必要な情報へ短く到達できるようにする。
- 下記の **冗長性の排除**、**構造化**、**厳密化** によって **解釈の不確実性の減少** と **読解コストの減少** を達成する。

## 冗長性の排除

### 内容の冗長性排除

- 同じ判断を別箇所から回復できる内容は削除候補にする。
- **削除候補**:
  - 同じ内容を複数箇所で述べている。
  - 文書の修正過程でのレガシー。
  - 要約内に具体手順、例、label、個別条件、可変値を含む。
  - 同一の概念に対する独立でない複数の例。
  - 見出しや parent item から回復できる主語、対象、条件を繰り返している。
  - 参照先に同じ規則がある。
  - 例外や補足が本則と同じ判断を別表現で述べている。
  - 移動、統合、抽象化の後に空になった構造が残っている。

### 文の冗長性排除

- 文の冗長性排除には **description item** が特に有効。
  - `:` の前後の句の統語的役割によって is, has / does / satisfies, consists of などを省略。
  - **value** は文や節である必要がなく、短く書くことが可能。
  - 下記の **構造の一貫性** に反しない範囲で **description item** を優先する。
- **例 (description item の型)**: 用法は以下に分類可能。
  - **同格型**: `名詞句 : 名詞句` : **label** is **value**
  - **述語型**: `名詞句 : 形容詞句 / 動詞句 / 節` : **label** has / does / satisfies **value**
  - **列挙型**: `名詞句 : List[句]` : **label** consists of `value1`, `value2`, ...
  - **文脈型**: `前置詞句 : 節` : In / under / for **label**, **value**
  - **条件型**: `条件節 : 節` : If / when **label**, then **value**
  - **フレーム型（record）**: `名詞句 : List[名詞句 : 句]` : **label** = { `field1` = `value1`, `field2` = `value2`, ... }

## 構造化

- **構造化**: 利用する source format の構造要素を使って、文書をスコープ単位で整理する。
  - ただし読みにくい細分化、機械的すぎる記法、markup の乱用は避ける。

### スコープへの分解

- **スコープ**: 文書内で意味境界を作る構造。section、list、environment、command、macro、record、schema、template、directory などを含む。
- スコープは、独立した責務、判断、契約のいずれかを担う。
- 大きな文書を、意味のあるスコープに分ける。
- 規模の大きなスコープ (sectionなど) では責務、判断、契約を明記する。

### スコープ関係

- **独立関係**: 互いに前後関係や参照関係を持たない。
  - **明示方法**: 両者が属する親スコープに両者を言及させる。
- **包含関係**: 一方のスコープがもう一方のスコープに含まれている。
  - **明示方法**: 親スコープが子スコープに言及。
- **依存関係**: 互いに包含関係はないが、前後関係、参照関係、前提関係を持つ。
  - **明示方法**: 両者が属する親スコープに両者を言及させるか、依存関係の方向に子スコープ同士で言及。

### スコープ責務の一般化

- スコープの責務、判断、契約には、特定タスクの値をハードコードしない。
- 会社名、日付、path、score、field value などの可変値は、入力、設定、template、schema、例として切り出す。
- スコープ本文には、具体例ではなく再利用できる責務、判断、契約を書く。

### Markup によるスコープ構造

- section、list、record などの markup 要素は、**スコープ** と **スコープ関係** を表すために使う。
- **独立関係** と **依存関係** は、まず共通の親スコープから記述する（**包含関係**）。
- **section**:
  - **解釈**: 名前を持つ大きなまとまり。独立した責務、関心、workflow phase、または順序や参照関係を持つまとまりを分ける。
  - **形式**: `Block[Title, Body]`
  - **関係**: 独立関係、依存関係。
  - **例**:
    ```markdown
    ## 調査

    対象、根拠、未確認事項をまとめる。
    ```

- **unordered list**:
  - **解釈**: 順序を持たない同型 item の集合。同じ型の item、選択肢、チェックリスト、分類を並べる。
  - **形式**: `Set[Item]`
  - **関係**: 独立関係。
  - **例**:
    ```markdown
    - 調査
    - 判断
    - 編集
    - 検証
    ```

- **ordered list**:
  - **解釈**: 順序または依存関係を持つ item の列。時系列、実行手順、優先順位、評価段階を並べる。
  - **形式**: `Seq[Step]`
  - **関係**: 依存関係。
  - **例**:
    ```markdown
    1. 調査する。
    2. 判断する。
    3. 編集する。
    4. 検証する。
    ```

- **ordered/unordered description**:
  - **解釈**: 各 step/item に record を付けたもの。
  - **形式**: `Seq/Set[Step/Item : Record]`
  - **関係**: 依存関係 / 独立関係
  - **例**:
    ```markdown
    1. **調査**:
       - **入力**: 対象文書
       - **出力**: 根拠一覧

    2. **判断**:
       - **入力**: 根拠一覧
       - **出力**: 修正方針
    ```

- **child list**:
  - **解釈**: parent item に従属する child item を持つ構造。parent item に従属する説明、条件、補足、例を置く。
  - **形式**: `Tree[Parent, Child]`
  - **関係**: 依存関係。
  - **例**:
    ```markdown
    - **規則**: ordered list は依存関係がある場合に使う。
      - **条件**: 前の step が後の step の前提になる。
      - **避ける例**: 独立した分類を ordered list にする。
    ```

- **ordered/unordered record**:
  - **解釈**: 複数対象を同じ field set で列挙または比較する構造。同じ種類の対象、同じ判断、同じ比較軸を並べる。
  - **形式**: `Set/Seq[Subject : Record[Field : Value]]`
  - **関係**: 独立関係、依存関係。
  - **例**:
    ```markdown
    - **target1**:
      - **field A**: value A
      - **field B**: value B
    - **target2**:
      - **field A**: value A
      - **field B**: value B
    ```

### 構造の一貫性

- **対象**: 同じ種類の対象、同じ判断、同じ比較軸、同じ workflow step、同じ入出力契約を複数回書く箇所。
- **基本規則**: 同じ意味のまとまりは、同じ文書構造、同じ見出し粒度、同じ item order、同じ label で書く。
- 使用可能な場合の各スコープの優先度:
  - **規模**: **section** と **child list** は比較不能。その都度適切に判断。
  - **一貫性の強さ**: **record** > **description** > **list**
  - **関係**: **ordered** > **unordered**
- **section**:
  - **不一致候補**: 見出し粒度、内部構造、目的、入力、出力、検証条件が揃っていない。
  - **正規化方針**: 同じ見出し粒度と同じ内部構造で書く。
- **list**:
  - **不一致候補**: unordered list、ordered list、description が混在している。並列 item の文法、粒度、終止形が揃っていない。
  - **正規化方針**: 順序や依存関係がある item だけ ordered list にし、並列 item の形を揃える。
- **description**:
  - **不一致候補**: 同じ役割の属性に異なる label を使っている。同じ label を異なる意味で使っている。label order が揃っていない。
  - **正規化方針**: 同じ役割には同じ label を使い、類似した description では label order を固定する。
- **record**:
  - **不一致候補**: 同じ判断に対する field set または field order が揃っていない。一部の対象だけ field を持たない。
  - **正規化方針**: 同じ field set/order を使う。必要な field は全対象に持たせ、`なし` のような `value` を許し無理に `value` を埋めない。
- **child list**:
  - **不一致候補**: 条件、補足、例、避ける例などの異なる役割が無標識に混ざっている。
  - **正規化方針**: child list に置く情報の役割を揃える。
- **列挙数**: 構造が一貫していても、列挙数が10を超えることはスコープの分割などで避ける。

## 厳密化

- **厳密化対象**:
  - 多用する概念とその用語割り当て。
  - **外部定義用語** と **内部定義用語** の分類。
  - 型、定義、実値、目印、本文を区別する名称。
  - スコープ由来の長い名称や重複。
- **揺れ候補**:
  - 同じ概念に複数の用語がある。
  - 同じ用語が複数の意味で使われている。
  - 外部形式または構造単位の英日表記が混在している。
  - 文書内の判断概念または操作名の英日表記が混在している。
  - 型、定義、実値、目印、本文の区別が名称から回復できない。

### 用語の厳密化

- 文書内で多用する概念には用語を割り当てる。
- 用語はプログラミング言語における変数や参照情報のように扱う。
  - 同じ用語には単一の意味/説明/定義を持たせる。
    - 抽象度の高い語を複数の概念に使い回さない。
    - 文脈ごとに意味が変わる用語にしない。
  - 同じ意味/説明/定義に単一の用語を持たせる。
    - 短縮形と正式名称を混ぜない。
    - 同一概念内で英日表記を混ぜない。
- 用語は外部定義用語と内部定義用語に分けられる。
- 外部形式または構造単位は主に英語で、文書内の判断概念と操作名は主に日本語を用いる。
- **外部定義用語**: 文書外のツールやドキュメントで定義された名称。
  - **対象**: code、path、field name、tool name、schema key、API 名、command、英語 documentation の terminology。
  - **表記**: 外部定義用語は code span で示す。
    - **例**: `README.md`
    - **例**: `name`
    - **例**: `git diff`
- **内部定義用語**: 文書内で定義された名称。
  - **対象**: section 名、概念名、責務名、判断基準名、workflow 名、step 名、入力名、出力名、schema 上の意味名、review 観点名、scoring 観点名。
  - **表記**: 内部定義用語は bold で示す。
    - **例**: **スコープ**
    - **例**: **スコープ関係**
    - **例**: **出力契約**

#### 用語の定義方法

- 内部定義用語は、文書内で使用される前にその定義、範囲、用法を明示する。
- 外部定義用語は文書内で再定義しない。
  - 定義元の名称と意味を維持する。
  - 外部名だけでは型や役割が曖昧な場合は、訳語や別名ではなく周辺説明で補う。
- 名称または説明名にはその型または意味に加え、必要なら接続関係、定義と実値、目印と内容の区別を含める。
- **外部定義用語の例**:
  - `placeholder` token: token / 後で値を差し込むための外部定義 marker。
  - `name` field key: schema key / 名称を保持する外部定義 field key。
  - `--output` CLI option: option / 出力先を指定する外部定義 command line option。
  - `git diff` command: command / 比較対象間の変更差分を出力する外部定義 command。
- **内部定義用語の例**:
  - **input slot definition**: definition / 入力値を受け取る枠の定義。
  - **input value instance**: value / **input slot definition** に渡される実際の値。
  - **comment anchor marker**: marker / コメントを付ける位置を示す目印。
  - **comment body text**: text / **comment anchor marker** に紐づけて表示する本文。
- 型を省いた名称は、受け口と差し込むもの、定義と実値、目印と内容、規則と結果を混同させる。
  - **悪い例**: **input**: 入力枠、入力値、入力 file、入力条件のどれか分からない。
  - **悪い例**: **placeholder**: 置換される token なのか、差し込む値なのか分からない。
  - **悪い例**: **comment**: コメント位置なのか、コメント本文なのか分からない。
  - **悪い例**: **field**: schema 上の field definition なのか、実際の field value なのか分からない。

#### スコープと用語

- スコープが明確な時、内部、外部の用語から、スコープ由来の単語を積極的に省く。
  - **例**: `テスト` section では単に `テストデータ集合` を `データ集合` と呼ぶ。
  - 用語だけで意味境界を分けない。
- スコープ名で同じ情報を重複させない。
- スコープの内容に対し、スコープの名前が適切かどうかを常に考慮する。

## 内容追加、レビュー、リバイズへの適用

- **内容追加**:
  - **目的**: 新しい内容を適切な **スコープ** に統合する。
  - **副作用**: 対象文書を書き換える。
- **レビュー検査**:
  - **目的**: 冗長性、構造、用語、情報保持の問題と修正候補を特定する。
  - **副作用**: 対象文書を書き換えない。
- **リバイズ**:
  - **目的**: **レビュー検査** と同じ確認を行い、対象文書を削除、統合、抽象化、移動、名称変更する。
  - **副作用**: 対象文書を書き換える。

### 内容追加前確認

- **適用条件**: 既存文書に新しい内容、条件、例、手順、field、record、section を追加する。
- **確認手順**:
  1. **追加内容の責務確認**: 追加内容が担う責務、判断、契約を特定する。
  2. **追加先スコープ確認**: 追加内容と同じ責務、判断、契約を持つ既存 **スコープ** を探す。
  3. **新規スコープ要否確認**: 既存 **スコープ** に入れると責務が混ざる場合だけ、新しい **スコープ** を作る。
  4. **抵触確認**: 追加内容が **冗長性の排除**、**構造化**、**厳密化**、**情報保持確認** に反しないか確認する。
  5. **統合確認**: 追加後に同じ判断を別箇所で重複させず、同じ種類の対象を同じ構造、同じ用語、同じ粒度で扱えているか確認する。
- **停止条件**: 追加先 **スコープ** または新規 **スコープ** 作成の要否を判断できない場合は、追加前に判断を求める。
- **出力**: 追加先 **スコープ**、新規 **スコープ** の要否、抵触確認の結果、情報保持確認の結果。

### 共通: スコープ単位確認

- **確認方向**: **葉スコープ** から **根スコープ** へ向かって確認する。
- **葉スコープ**: 子スコープを持たない最小の意味単位。
- **親スコープ**: 子スコープの **スコープ関係**、要約、共通規則を受け持つ **スコープ**。
- **根スコープ**: 対象文書の最上位 **スコープ**。
- **確認内容**:
  - **葉スコープ**: 冗長性、構造、用語、情報保持の問題。
  - **親スコープ**: 子スコープ間の **スコープ関係**、**役割重複**、**構造の一貫性**、共通規則、要約の問題。
  - **根スコープ**: 文書全体の目的、責務、参照案内、出力契約の問題。
- **確認手順**:
  1. **スコープ木の作成**: **根スコープ**、**親スコープ**、**葉スコープ** を列挙する。
  2. **処理順序の固定**: **葉スコープ** を先に並べ、各 **親スコープ** はすべての子スコープを処理した後に扱う。
  3. **葉スコープの確認**: **葉スコープ** の冗長性、構造、用語、情報保持を確認する。
    - リバイズ時は実際に修正を行う。
  4. **葉スコープの分離確認**: **葉スコープ** の分離を検討する。
  5. **親スコープの確認**: 子スコープ間の **スコープ関係**、**役割重複**、**構造の一貫性**、共通規則、要約を確認する。
    - リバイズ時は実際に修正を行う。
  6. **兄弟スコープ統合確認**: 同じ親を持つ子スコープの役割が重なる場合は、統合、親スコープへの引き上げ、または役割分離のいずれかにできるか確認する。
    - リバイズ時は実際に修正を行う。
  7. **親スコープへの反映候補確認**: 共通規則、要約、参照案内が **親スコープ** にあることを確認する。
    - リバイズ時は実際に修正を行う。
  8. **上位反復**: **根スコープ** に到達するまで、処理済み **親スコープ** を子スコープとみなして同じ手順を繰り返す。
- **反復条件**:
  - **レビュー検査**: **確認手順** は一度だけ行う。
  - **リバイズ**: **確認手順** で修正が行われれば、もう一度 **確認手順** を繰り返す。
- **停止条件**:
  - この rule だけで判断できない問題は処理を停止し、ユーザーに判断を求める。

### 副作用と出力

- **レビュー検査**:
  - **副作用**: 対象文書を書き換えない。
  - **出力**: スコープ単位の finding、根拠、最小修正案。修正案による情報保持の確認の結果。
- **リバイズ**:
  - **副作用**: 対象文書を書き換える。
  - **出力**: 変更したスコープ、変更内容、情報保持確認の結果。

### 例外

- 明確な子スコープがない短い文書では、文書全体を 1 つの **葉スコープ** として扱う。
- **葉スコープ** の修正で **親スコープ** の契約または **スコープ関係** が変わる場合は、修正前に影響を明示する。

### 情報保持確認

- **適用条件**: 削除、統合、抽象化、移動、名称変更などを検討または実行する。
- **レビュー検査**: 修正候補の適用後も、必要な判断を **根スコープ** から **葉スコープ** まで再現できるか確認する。
- **リバイズ**: 書き換え後も、必要な判断を **根スコープ** から **葉スコープ** まで再現できるか確認する。
- **修正方針**: 必要な判断を再現できない内容は、抽象規則、定義、根拠、または参照案内として残す。
