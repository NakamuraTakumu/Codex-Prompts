# Structured Document Rules

## 用途

- **目的**: **構造化文書** を簡潔に保ち、人間と LLM が同じ構造、用語、参照関係をたどれるようにする。
- **使用場面**: 構造化文書の作成、内容変更、レビュー、リバイズで、冗長性、構造、用語・表記、情報保持を判断する。
- **構造化文書**: Markdown、LaTeX、その他の markup または source format で書かれ、section、list、environment、command、macro、record、schema などで意味境界を表す文書。
- **対象例**: `SKILL.md`、bundled resources、プロンプト本文、仕様書、手順書、レビュー文書、LaTeX 原稿。
- **自己適用**: この file 自体も **構造化**、**冗長性の排除**、**厳密化**、**情報保持** の適用例として参照できる。

## 実行規則

- ユーザー要求を **新規作成**、**内容変更**、**レビュー検査**、**リバイズ** に分類し、**文書作業規則** と **文書作業への適用** に従う。
- ファイルを書き換える操作は、変更方針を明示し、ユーザーの明示的な編集依頼または許可がある場合だけ行う。
- 作業種別を判断できない場合は、目的、対象読者、入力、出力契約、初期スコープ構造のうち不足する情報を確認する。

## 用語

### Description と Record

- **field**: **description item** または **record** で、意味上の属性名または比較軸として使う名称。
- **label**: markup 上に現れる **field** の表示名。**field** と異なる語を使う場合は対応関係を明示する。
- **value**: **field** に対応する内容。markup 上では **label** に続けて書く。
- **description item**: **field** に対応する **label** と **value** の組で、対象、条件、例などの属性を示す item。
  - **例**: `**label**: value`
- **description**: 1 つの対象に対する description item の集合。
- **list**: field を持たない同型 item の列挙。
- **field set**: 同じ対象群に対して使う field の集合。
- **field order**: 同じ field set の field を並べる順序。
- **unordered description**: description item を順序なしで並べたもの。
- **ordered description**: description item を順序付きで並べたもの。
- **record**: 複数対象を同じ **field set** で扱うとき、各対象に同じ **description** を並べた構造。

### 言語単位

- **言語単位**: 文法上の単位。Markdown の構造名ではなく、文、節、句などを指す。
- **文**: 完結した記述単位。
- **節**: 述語または主述関係を含む言語単位。文の一部になる場合と、文として独立する場合がある。
- **句**: 文や節より小さい語句のまとまり。名詞句、形容詞句、動詞句、前置詞句など。

### スコープ単位

- **スコープ**: 責務、判断、契約、参照関係の境界を作る単位。**文書スコープ**、**構文スコープ**、**配置スコープ** に分ける。
- **文書スコープ**: section、段落、list、environment など、文書内の意味境界。
- **構文スコープ**: command、macro、record、schema、template など、source format 上の境界。
- **配置スコープ**: file、directory など、文書外の配置境界。
- **葉スコープ**: 子スコープを持たない最小の意味単位。
- **親スコープ**: 子スコープの **スコープ関係**、要約、共通規則を受け持つ **スコープ**。
- **根スコープ**: 対象文書の最上位 **スコープ**。
- **処理単位**: 明確な子スコープがない短い文書では、文書全体を 1 つの **葉スコープ** として扱う。
- **スコープ木**: **根スコープ**、**親スコープ**、**葉スコープ** の包含関係を列挙した構造。
- **基準入力**: 使用者、呼び出し元、対象文書、関連 file が明示または参照するレビュー基準、リバイズ基準、style guide、checklist、出力契約。

## 基本原則

- 基本原則は、解釈、読解、情報保持、構造選択を同時に満たす判断基準として使う。
- **解釈の不確実性の減少**: 対象、目的、前提、定義、主張、根拠、結論、責務境界を明示し、同じ語を同じ意味で使う。
- **読解コストの減少**: 読む順序、階層、比較軸、例と本則の関係を文書構造で示し、必要な情報へ短く到達できるようにする。
- **情報保持**: 削除、統合、抽象化、移動、名称変更の後も、必要な判断、契約、根拠、参照関係を再現できるようにする。
- **構造選択**: markup 化を目的にせず、責務、判断、契約、参照関係を最も短く安定して表せる構造を選ぶ。
- **適用軸**: **冗長性の排除**、**構造化**、**厳密化**、**情報保持** を組み合わせて判断する。

## 冗長性の排除

### 内容の冗長性排除

- 同じ判断を別箇所から回復できる内容は削除候補にする。
- **削除候補**:
  - 同じ内容を複数箇所で述べている。
  - 文書の修正過程でのレガシー。
  - 要約内に具体手順、例、label、個別条件、可変値を含む。
  - 同一の概念に対する独立でない複数の例。
  - 見出しや parent item から回復できる主語、対象、条件を繰り返している。
  - 参照先に同じ規則がある。
  - 例外や補足が本則と同じ判断を別表現で述べている。
  - 移動、統合、抽象化の後に空になった構造が残っている。
- **保持候補**:
  - 削ると入力、判断条件、禁止事項、出力形式、検証条件、副作用条件、失敗時の挙動を再現できない。
  - 例や補足が判断差、fallback、例外処理、tool selection を固定している。

### 文の冗長性排除

- **基本方針**: 同じ種類の対象に属性、条件、結果、例を付ける場合は、**構造の一貫性** に反しない範囲で **description item** を優先する。
- **効果**: `:` の前後の統語的役割によって is, has / does / satisfies, consists of などを省略できる。
- **適用制限**: 因果、対比、例外、手順移行、論証を表す文は、句へ圧縮せず **自然言語によるスコープ構造** で関係を明示する。
- **value**: 文や節である必要はない。句で同じ判断を表せる場合は句にする。
- **変換型**:
  - **同格型**: `A(名詞句) は B(名詞句) である` -> `A: B`
  - **述語型**: `A(名詞句) は B(形容詞句 / 動詞句 / 節) である / する / 満たす` -> `A: B`
  - **列挙型**: `A(名詞句) は B1(句), B2(句), ... からなる` -> `A: B1, B2, ...`
  - **文脈型**: `A(前置詞句) では / のもとでは / について、B(節)` -> `A: B`
  - **条件型**: `A(条件節) ならば、B(節)` -> `A: B`
  - **フレーム型 (record)**: `A(名詞句) について、F1(名詞句) は B1(句)、F2(名詞句) は B2(句) である` -> `A: { F1: B1, F2: B2 }`

## 構造化

この section では、**スコープへの分解**、**スコープ関係**、**スコープ責務の一般化**、**Markup によるスコープ構造**、**自然言語によるスコープ構造**、**構造の一貫性** を扱う。

- **構造化**: 利用する source format の markup 要素と自然言語を使って、文書をスコープ単位で整理する。
  - ただし読みにくい細分化、機械的すぎる記法、markup の乱用は避ける。

### スコープへの分解

- スコープは、独立した責務、判断、契約のいずれかを担う。
- 大きなスコープを、意味のある小さなスコープに分ける。
- 規模の大きなスコープ (sectionなど) では責務、判断、契約を明記する。

### スコープ関係

- **親スコープ冒頭**: 親スコープは、本文の最初に保持する子スコープの名前と役割に言及する。
- **参照導線**: 使用場面、作業種別、workflow、確認規則、判断基準から使われるスコープには、使用元からスコープ名を明示して参照を伸ばす。
- **独立関係**: 互いに前後関係や参照関係を持たない。
  - **明示方法**: 両者が属する親スコープに両者を言及させる。
- **包含関係**: 一方のスコープがもう一方のスコープに含まれている。
  - **明示方法**: 親スコープが子スコープに言及する。
- **依存関係**: 互いに包含関係はないが、前後関係、参照関係、前提関係を持つ。
  - **明示方法**: 両者が属する親スコープに両者を言及させるか、依存方向に子スコープ同士で言及する。

### スコープ責務の一般化

- スコープの責務、判断、契約には、特定タスクの値をハードコードしない。
- 会社名、日付、path、score、field value などの可変値は、入力、設定、template、schema、例として切り出す。
- スコープ本文には、本則として再利用できる責務、判断、契約を書く。具体値が判断固定に必要な場合は、例として切り出す。

### Markup によるスコープ構造

- section、list、record などの markup 要素は、**スコープ** と **スコープ関係** を表すために使う。
- **独立関係** と **依存関係** は、まず共通の親スコープから記述する（**包含関係**）。
- markup を増やしても責務、判断、契約、参照関係が明確にならない場合は、段落内の自然言語で関係を示す。
- **選択規則**:
  - **意味境界**: まとまりに名前、責務、参照関係が必要なら **section** を使う。
  - **比較軸**: 同じ field set で複数対象を扱うなら **record** を使う。
  - **属性付け**: item ごとに属性を付けるなら **description** を使う。
  - **description の順序**: field order で読む順序や比較軸を示す場合は **ordered description**、順序に判断がない場合は **unordered description** を使う。
  - **従属説明**: parent item に条件、補足、例を置くなら **child list** を使う。
  - **列挙**: 同型 item を並べるだけなら **list** を使う。
  - **関係**: 順序または依存関係がある場合は **ordered**、ない場合は **unordered** を使う。

### 自然言語によるスコープ構造

- 自然言語は、markup に切り出すほどではない境界、前後関係、参照関係、責務境界を示すために使う。
- **適用条件**:
  - markup を追加すると文書が細分化されすぎる。
  - 段落内または item 内で、前提、主張、根拠、結論、例外、補足の役割を区別する。
  - 兄弟スコープ間または親子スコープ間の関係を、見出しや list だけでは回復しにくい。
- **境界表現**:
  - **topic sentence**: 段落または item の先頭で、そのスコープの責務、判断、契約を述べる。
  - **preview**: 複数の子スコープを導入する前に、読む順序、比較軸、列挙対象を述べる。
  - **summary**: 子スコープ群の後で、共通判断、結論、未解決事項を述べる。
  - **transition**: 前後のスコープの関係が変わる箇所で、追加、対比、例外、帰結、手順移行を述べる。
  - **reference phrase**: 参照先、根拠、前提、依存先を明示する。
- **関係表現**:
  - **包含関係**: 「以下では」「この section では」「この item は」などで親スコープを示す。
  - **依存関係**: 「この前提のもとで」「上記を受けて」「その後」「したがって」などで前後関係を示す。
  - **独立関係**: 「別の観点として」「一方で」「独立に」などで同列だが依存しないことを示す。
  - **例外表現**: 「ただし」「例外として」「この場合だけ」などで本則との差分を示す。

### 構造の一貫性

- **基本規則**: 同じ親に属する並列の言語単位または子スコープは、形式、意味的役割、粒度を揃える。
- **優先対象**: 表面的な形より、責務、判断、契約、参照関係の対応を優先して揃える。
- **優先規則**: 一貫性を保てる場合は **record**、**description**、**list** の順に、判断軸を明示しやすい構造を選ぶ。

#### 言語単位の一貫性

- **文**:
  - **不一致候補**: 並列する文の主語、述語、時制、モダリティ、終止形、意味的役割が揃っていない。
  - **正規化方針**: 同じ役割の文は同じ構文型、同じ終止形、同じ粒度で書く。
- **節**:
  - **不一致候補**: 条件、理由、目的、例外、帰結などの節が無標識に混ざっている。
  - **正規化方針**: 同じ役割の節には同じ接続表現を使い、異なる役割の節は分ける。
- **句**:
  - **不一致候補**: 名詞句、動詞句、形容詞句などの構文型が並列内で混ざっている。
  - **正規化方針**: 並列する句の構文型、抽象度、長さを揃える。
- **段落**:
  - **不一致候補**: 段落の役割、内部順序、またはスコープ関係が揃っていない。段落で示した関係が markup 構造と矛盾している。
  - **正規化方針**: 同じ役割の段落は同じ内部順序で書き、スコープ関係を明示する。

#### 子スコープの一貫性

- **section**:
  - **不一致候補**: 見出し粒度、内部構造、目的、入力、出力、検証条件が揃っていない。
  - **正規化方針**: 同じ見出し粒度と同じ内部構造で書く。
- **list**:
  - **不一致候補**: unordered list、ordered list、description が混在している。並列 item の文法、粒度、終止形が揃っていない。
  - **正規化方針**: 順序や依存関係がある item だけ ordered list にし、並列 item の形を揃える。
- **description**:
  - **不一致候補**: 同じ役割の属性に異なる field を使っている。同じ field を異なる意味で使っている。field order が揃っていない。
  - **正規化方針**: 同じ役割には同じ field を使い、類似した description では field order を固定する。
- **record**:
  - **不一致候補**: 同じ判断に対する field set または field order が揃っていない。一部の対象だけ field を持たない。
  - **正規化方針**: 同じ field set/order を使う。必要な field は全対象に持たせ、`なし` のような `value` を許し無理に `value` を埋めない。
- **child list**:
  - **不一致候補**: 条件、補足、例、避ける例などの異なる役割が無標識に混ざっている。
  - **正規化方針**: child list に置く情報の役割を揃える。
- **列挙数**: 構造が一貫していても、列挙数が 10 を超える場合はスコープ分割を検討する。

## 厳密化

- この section では、用語分類、表記、定義位置、名称の粒度を扱う。
- **厳密化対象**:
  - 多用する概念とその用語割り当て。
  - **外部定義用語**、**内部定義用語**、**一般用語** の分類。
  - 型、定義、実値、目印、本文を区別する名称。
  - スコープ由来の長い名称や重複。
- **揺れ候補**:
  - 同じ概念に複数の用語がある。
  - 同じ用語が複数の意味で使われている。
  - 外部形式、一般用語、構造単位の英日表記が混在している。
  - 文書内の判断概念または操作名の英日表記が混在している。
  - 型、定義、実値、目印、本文の区別が名称から回復できない。

### 用語の厳密化

- 文書内で多用する概念には用語を割り当てる。
- 用語はプログラミング言語における変数や参照情報のように扱う。
  - 同じ用語には単一の意味/説明/定義を持たせる。
    - 抽象度の高い語を複数の概念に使い回さない。
    - 文脈ごとに意味が変わる用語にしない。
  - 同じ意味/説明/定義に単一の用語を持たせる。
    - 短縮形と正式名称を混ぜない。
    - 同一概念内で英日表記を混ぜない。
- 用語は **外部定義用語**、**内部定義用語**、**一般用語** に分けられる。
- 外部形式と構造単位の **一般用語** は主に英語で、文書内の判断概念と操作名は主に日本語を用いる。
- **外部定義用語**:
  - **対象**: code、path、field name、tool name、schema key、API 名、command、英語 documentation の terminology。
  - **表記**: 外部定義用語は code span で示す。
- **内部定義用語**:
  - **対象**: section 名、概念名、責務名、判断基準名、workflow 名、step 名、入力名、出力名、schema 上の意味名、review 観点名、scoring 観点名。
  - **表記**: 内部定義用語は bold で示す。
- **一般用語**:
  - **対象**: section、list、record、description、field など、文書外でも一般に使われ、この文書で用法を限定する語。
  - **表記**: 通常の本文表記を使う。用法を固定したい箇所だけ bold で示す。

#### 用語の定義方法

- 内部定義用語は、文書内で使用される前にその定義、範囲、用法を明示する。
- 用語の定義は、その用語の全使用箇所を含む最小の **スコープ** に置く。
  - **広すぎる定義位置**: 使用範囲より広い **スコープ** に置いて、局所用語を文書全体の用語にする。
  - **狭すぎる定義位置**: 使用箇所の一部から参照できない下位 **スコープ** に置く。
- 外部定義用語は文書内で再定義しない。
  - 定義元の名称と意味を維持する。
  - 外部名だけでは型や役割が曖昧な場合は、訳語や別名ではなく周辺説明で補う。
- 名称または説明名にはその型または意味に加え、必要なら接続関係、定義と実値、目印と内容の区別を含める。
- **外部定義用語の例**:
  - `name` field key: schema key / 名称を保持する外部定義 field key。
  - `--output` CLI option: option / 出力先を指定する外部定義 command line option。
- **内部定義用語の例**:
  - **input slot definition**: definition / 入力値を受け取る枠の定義。
  - **input value instance**: value / **input slot definition** に渡される実際の値。
- 型を省いた名称は、受け口と差し込むもの、定義と実値、目印と内容、規則と結果を混同させる。
  - **悪い例**: **input**: 入力枠、入力値、入力 file、入力条件のどれか分からない。
  - **悪い例**: **placeholder**: 置換される token なのか、差し込む値なのか分からない。

#### スコープと用語

- スコープが明確な時、内部、外部の用語から、スコープ由来の単語を積極的に省く。
- 用語だけで意味境界を分けない。
- スコープ名で同じ情報を重複させない。
- スコープの内容に対し、スコープの名前が適切かどうかを常に考慮する。

## 文書作業規則

この section では、文書作業の作業種別、分類規則、副作用規則、基準入力の扱い、作業前確認規則、**スコープ構造の確認規則**、不足情報規則を定める。

- **作業種別**:
  - **新規作成**: 目的、対象読者、入力、出力、初期スコープ構造を設定して新しい文書を作る。
  - **内容変更**: 既存文書の意味、判断、契約、根拠、参照関係、出力契約を追加、削除、置換、更新する。
  - **レビュー検査**: 冗長性、構造、用語、情報保持の問題と修正候補を特定し、対象文書を書き換えない。
  - **リバイズ**: **レビュー検査** と同じ確認を行い、意味や内容の変化を伴わない **内容変更** として対象文書を書き換える。
- **分類規則**:
  - **意味変化あり**: 既存文書から回復できない情報、判断、契約、根拠、参照関係、出力契約の増減または置換を伴う場合は **内容変更** として扱う。
  - **意味保持**: 削除、統合、抽象化、移動、名称変更、表記統一、スコープ構造修正、構造整理が意味や内容の変化を伴わない場合は **リバイズ** として扱う。
  - **副作用なし**: 対象文書を書き換えず finding または修正候補だけを出す場合は **レビュー検査** として扱う。
- **副作用規則**:
  - **新規作成**: 新しい文書を作る。
  - **内容変更** と **リバイズ**: 対象文書を書き換える。
  - **レビュー検査**: 対象文書を書き換えない。
- **基準入力の扱い**:
  - **扱い**: 基準入力は **スコープ構造検査** と **情報保持確認** で併用する。
  - **衝突時**: 基準入力がこの rule の情報保持、副作用規則、または上位指示と衝突する場合は、編集前に衝突を明示し、判断を求める。
- **作業前確認規則**:
  - **新規作成**: 目的、対象読者、入力、出力契約、初期スコープ構造、未確定事項を特定する。
  - **内容変更**: 変更意図、変更先 **スコープ**、新規 **スコープ** の要否、抵触の有無、統合後の一貫性、未確定事項を特定する。
  - **抵触確認**: 変更内容が **冗長性の排除**、**構造化**、**厳密化** に反しないか確認し、変更対象外の情報について **情報保持確認** を行う。
- **不足情報規則**: 一般文書では、判断できない情報を仮置きまたは未確定事項として明示する。指示文書の入力、判断、出力、検証、副作用条件に影響する場合は `skills/common/instruction_rule.md` に従う。

### スコープ構造の確認規則

- **確認対象**:
  - **葉スコープ**: 冗長性、構造、用語、情報保持、**スコープ名**。
  - **親スコープ**: 子スコープ間の **スコープ関係**、**参照導線**、**役割重複**、**構造の一貫性**、共通規則、要約。
  - **根スコープ**: 文書全体の目的、責務、参照案内、出力契約、**参照導線**。
  - **読みにくい箇所**: 責務、判断、契約、読む順序、参照関係を追いにくい **スコープ**。
- **修正規則**:
  - **境界修正**: 責務が複数ある **スコープ** は分離し、独立しない細片は統合する。
  - **名称修正**: **スコープ名** は責務、判断、契約を表し、親や子の情報を不要に繰り返さない。
  - **構造修正**: 責務、判断、契約、参照関係を最短で追えない場合は、分割、統合、移動、引き上げを行う。
  - **到達可能性**: 使用場面、作業種別、workflow、確認規則、判断基準から使うべき **スコープ** へ、スコープ名で到達できるようにする。
- **横断規則**:
  - **横断確認**: 離れた **スコープ** 間で、同じ用語、判断、入力、出力契約、副作用条件、参照関係を矛盾または重複させない。
  - **契約追跡**: **根スコープ** の目的、入力、出力契約、参照案内を、必要な **葉スコープ** まで具体化する。

## 文書作業への適用

- **適用手順**:
  - **新規作成**: **新規作成前確認**、初期作成、**リバイズ対象範囲** を対象にした **リバイズ** の順に行う。
  - **内容変更**: **内容変更前確認**、変更適用、**リバイズ対象範囲** を対象にした **リバイズ** の順に行う。
  - **レビュー検査** と **リバイズ**: **スコープ構造検査** と **情報保持確認** に従う。
- **リバイズ対象範囲**:
  - **基準**: 新規作成または内容変更で作成、変更、削除、移動した各 **スコープ** を **変更スコープ** とする。
  - **最小範囲**: 各 **変更スコープ** から二段階上がった **スコープ** 全体を対象にする。二段階上がった **スコープ** が存在しない場合は **根スコープ** 全体を対象にする。
  - **拡張条件**: **変更スコープ** の参照関係、契約、用語、出力、副作用が最小範囲の外側に影響する場合は、影響先まで対象を広げる。

### 新規作成前確認

- **確認**: **作業前確認規則** の **新規作成** を満たす。
- **初期スコープ設計**: **スコープへの分解**、**スコープ関係**、**Markup によるスコープ構造**、**自然言語によるスコープ構造**、**構造の一貫性** に従う。
- **出力**: 目的、対象読者、入力、出力契約、初期スコープ構造、未確定事項。

### 内容変更前確認

- **確認**: **作業前確認規則** の **内容変更** を満たす。
- **配置**: 変更内容と同じ責務、判断、契約を持つ既存 **スコープ** に入れ、責務が混ざる場合だけ新しい **スコープ** を作る。
- **出力**: 変更意図、変更先 **スコープ**、新規 **スコープ** の要否、抵触確認の結果、変更対象外の情報保持確認の結果、未確定事項。

### スコープ構造検査

- **準備手順**:
  1. **基準入力の特定**: **基準入力** を特定する。
  2. **スコープ木の作成**: **根スコープ**、**親スコープ**、**葉スコープ** を列挙する。
  3. **読みにくい箇所の特定**: 責務、判断、契約、読む順序、参照関係を追いにくい **スコープ** を優先対象にする。
  4. **処理順序の固定**: **葉スコープ** を先に並べ、各 **親スコープ** はすべての子スコープを処理した後に扱う。**読みにくい箇所** は同じ階層内で先に扱う。
- **確認規則**: **確認手順** の各 step は、**基準入力**、**スコープ構造の確認規則**、この rule の基準を併用して判断する。
- **確認手順**:
  1. **葉スコープの確認**: **葉スコープ** に対して **スコープ構造の確認規則** の **葉スコープ** を適用する。リバイズ時は修正する。
  2. **葉スコープの境界確認**: **スコープ構造の確認規則** の **境界修正** を適用する。リバイズ時は修正する。
  3. **スコープ名確認**: **スコープ構造の確認規則** の **名称修正** を適用する。リバイズ時は修正する。
  4. **親スコープの確認**: **親スコープ** に対して **スコープ構造の確認規則** の **親スコープ** を適用する。リバイズ時は修正する。
  5. **スコープ構造確認**: **スコープ構造の確認規則** の **構造修正** を適用する。リバイズ時は修正する。
  6. **到達可能性確認**: **スコープ構造の確認規則** の **到達可能性** を適用する。リバイズ時は参照元に明示参照を追加する。
  7. **親スコープへの反映確認**: 共通規則、要約、参照案内が必要な **親スコープ** にあるか確認する。リバイズ時は修正する。
  8. **上位反復**: **根スコープ** に到達するまで、処理済み **親スコープ** を子スコープとみなして同じ手順を繰り返す。
  9. **横断確認**: **スコープ構造の確認規則** の **横断確認** を適用する。リバイズ時は修正する。
  10. **契約追跡確認**: **スコープ構造の確認規則** の **契約追跡** を適用する。リバイズ時は修正する。
- **反復条件**:
  - **レビュー検査**: **準備手順** と **確認手順** は一度だけ行う。
  - **リバイズ**: **確認手順** で修正が行われれば、**準備手順** から繰り返す。
- **不足情報の扱い**: 上記の **不足情報規則** に従う。

### 情報保持確認

- **適用条件**: 削除、統合、抽象化、移動、名称変更などを検討または実行する。
- **確認問い**: 修正候補の適用後または書き換え後も、必要な判断を **根スコープ** から **葉スコープ** まで再現できるか。
- **保持対象**: 入力、判断条件、禁止事項、出力形式、検証条件、副作用条件、失敗時の挙動、根拠、参照案内。
- **修正方針**: 必要な判断を再現できない内容は、抽象規則、定義、根拠、schema、template、または参照案内として残す。

### 副作用と出力

- **レビュー検査**:
  - **副作用**: 対象文書を書き換えない。
  - **出力**: スコープ単位の finding、根拠、最小修正案、情報保持確認の結果。
- **リバイズ**:
  - **副作用**: 対象文書を書き換える。
  - **出力**: 変更したスコープ、変更内容、情報保持確認の結果。
- **例外**:
  - **葉スコープ** の修正で **親スコープ** の契約または **スコープ関係** が変わる場合は、修正前に影響を明示する。
