# Instruction Rules

## 用途

- **指示文書** を lean に保つための特化規則。
- **指示文書** の作成、更新、レビュー、リバイズで、文書構造、冗長性、用語・表記、情報保持の判断が必要な場合は、先に `skills/common/structured_document_rule.md` を読む。
- **指示文書** の入力契約、副作用契約、選択規則、失敗時の挙動、完了条件を厳密化する場合に読む。
- **指示文書**: `SKILL.md`、bundled resources、プロンプト本文など、Codex の動作を規定する文書。

## 参照

- **`skills/common/structured_document_rule.md`**: 構造化文書に共通する冗長性、構造化、情報保持、用語の厳密化の規則。

## 構造化文書規則の指示文書への適用

- **`基本原則`**:
  - **解釈の不確実性の減少**: 指示文書では、同じ入力、同じ状態、同じ参照情報に対して、同じ判断、同じ操作、同じ報告になることとして読む。
    - 指示の矛盾は解釈の不確実性を確実に悪化させる。
      - 特に相反する目標を二つ立ててバランスを取らせるときは注意。
  - **読解コストの減少**: 指示文書では、読むべき情報、使うべき tool、確認すべき条件、停止条件、出力形式へ短く到達できることとして読む。
- **`冗長性の排除`**:
  - `skills/common/structured_document_rule.md` の削除候補条件を使う前に、削除対象が **入力**、**手順**、**判断条件**、**禁止事項**、**出力契約**、**検証条件**、**副作用条件** を変えないか確認する。
  - 同じ判断条件、禁止事項、手順、例が別箇所から同じ粒度で回復できる場合は、削除または一元化する。
  - 特定タスクの一回限りの値は、再利用可能な規則として書かず、必要に応じて入力、設定、template、schema、例として切り出す。
  - 例や補足を削っても同じ判断を再現できる場合は削除候補にする。
  - 一般文書では冗長に見える例や補足でも、指示文書の判断差、fallback、例外処理、tool selection を固定する場合は残す。
- **`構造化`**:
  - **`スコープへの分解`**: 指示文書では、目的、入力契約、手順、判断基準、副作用契約、出力契約、検証条件を別スコープとして扱う。
  - **`スコープ関係`**: 手順の前後関係、参照 file の依存関係、承認条件、fallback の分岐を明示する。
  - **`スコープ責務の一般化`**: 具体 path、command、schema key、model 名、score などは、固定値として必要な場合を除き、入力、設定、template、例として切り出す。
  - **`Markup によるスコープ構造`** と **`構造の一貫性`**: 同じ種類の入力、手順、判断、出力、検証は同じ見出し粒度、同じ field set、同じ field order で書く。
- **`情報保持確認`**:
  - 削除、統合、抽象化、移動の後も、必要な入力、判断条件、禁止事項、出力形式、検証条件、副作用条件、失敗時の挙動を再現できるか確認する。
  - 回復できない具体例は、抽象規則、契約、schema、template、または参照案内として残す。
- **`厳密化`** と **`用語の厳密化`**:
  - 外部定義用語は再定義せず、tool name、command、field name、schema key、API 名、file path の表記を維持する。
  - 内部定義用語は、入力枠と入力値、定義と実値、目印と本文、規則と結果が混同されないように定義する。
- **`内容追加、レビュー、リバイズへの適用`**:
  - 内容追加では、`skills/common/structured_document_rule.md` の **内容追加前確認** で追加先 **スコープ** と抵触の有無を先に確認する。
  - レビューまたはリバイズでは、`skills/common/structured_document_rule.md` の **共通: スコープ単位確認** と **情報保持確認** で文書としての確認を先に行う。
  - その後にこの file の **指示文書固有の確認** で、指示文書固有の入力、手順、出力、検証、副作用を確認する。

## 指示文書の記述範囲

- **指示文書** として記述するのは、基本的に入力、手順、出力、副作用条件の指定。
  - **入力**: 参照する情報、受け取る値、前提、不明情報の扱い。
  - **手順**: 調査、判断、変換、編集、レビューなどの進め方。
  - **出力**: 形式、粒度、保存先、返答内容、schema、検証条件。
  - **副作用条件**: ファイル編集、生成、削除、command 実行、sub-agent 起動などの実行条件。
- **指示文書** を記述する目的は、動作の不確実性の減少と、読解コストの減少。
  - **動作の不確実性の減少**: 判断基準、責務境界、禁止事項、出力契約、副作用条件の明示。
  - **読解コストの減少**: 読むべき情報、使うべき tool、参照すべき file、確認すべき条件の明示。

## 指示文書固有の確認

- **冗長性確認**:
  - **確認問い**: その記述は入力、手順、判断、出力、検証、副作用のいずれかを変えるか。別の記述から同じ粒度で回復できないか。
  - **修正方針**: 不要なら削除し、必要なら要約と詳細、抽象原則と具体規則に分ける。
- **構造化確認**:
  - **確認問い**: 見出し、list、手順、補助資料の分担は内容の責務と一致するか。役割が重なる子スコープを統合、親スコープへの引き上げ、または役割分離できないか。読む順序、分岐、停止条件が誤解なく追えるか。
  - **修正方針**: 責務を分離し、一元化し、同じ種類の対象は揃った構造で表す。
- **厳密化確認**:
  - **確認問い**: 名称、表記、参照、入力契約、副作用、選択規則、失敗時の挙動、完了条件に曖昧さや揺れがないか。
  - **修正方針**: 意味境界を明確にし、同じ意味には同じ名称、異なる意味には異なる名称を使う。
- **情報保持確認**:
  - **確認問い**: 削除、統合、抽象化、移動の後も、必要な入力、判断、禁止事項、出力、検証、副作用条件を再現できるか。
  - **修正方針**: 具体例や詳細を削る場合は、同じ判断を再現できる抽象規則、契約、schema、template、または参照案内を残す。

## 内容追加、レビュー、リバイズへの適用

- **優先順位**: 冗長性の排除、構造化、厳密化、情報保持を最優先し、実動作の改善はこれらを損なわない場合だけ採用する。
- **内容追加**:
  - `skills/common/structured_document_rule.md` の **内容追加前確認** に従う。
  - 追加内容を入れるべき **スコープ** を確認し、冗長性、構造、厳密化、情報保持への抵触がない場合だけ追加する。
- **レビューとリバイズ**:
  - `skills/common/structured_document_rule.md` の **共通: スコープ単位確認** の **確認手順** と **反復条件** に従う。
  - 文書としてのレビューまたはリバイズを先に行い、その後に指示文書としてのレビューまたはリバイズを行う。
- **レビュー検査**: 文書としての **確認手順** を一度だけ行った後、**指示文書固有の確認** を一度だけ行い、finding を出す。
- **リバイズ**:
  - 文書としての **確認手順** の中で対象文書を修正した場合は、もう一度 **確認手順** を繰り返す。
  - 文書としての修正箇所がなくなった後、**指示文書固有の確認** に従って修正する。
  - 指示文書固有の修正を行った場合は、文書としての **確認手順** に戻る。
  - 指示文書固有の修正箇所がなくなるまで、文書としての **確認手順** と **指示文書固有の確認** を繰り返す。
- **完了条件**: 文書としての修正箇所がなく、かつ指示文書固有の入力、手順、判断条件、禁止事項、出力契約、検証条件、副作用条件に関する修正箇所がない。
- **停止条件**: この rule と `skills/common/structured_document_rule.md` だけで判断できない問題は処理を停止し、ユーザーに判断を求める。

## プロンプトの挙動の厳密化

- 挙動の厳密化は、文書品質と契約保持に従属する。
- 同じ入力、同じ状態、同じ参照情報では、同じ判断と同じ副作用になるように書く。
- 特に動作するシステムに干渉させる指示は一層厳密化する。

### 契約の明示

- **入力契約**:
  - **対象**: 必須入力、任意入力、既定値、不明情報。
  - **書くこと**: 何を受け取り、何がない場合に質問、仮定、中断のどれを選ぶか。
  - **避けること**: 入力不足時の扱いを workflow の推測に任せる。
- **副作用契約**:
  - **対象**: ファイル読み書き、生成、削除、command 実行、sub-agent 起動。
  - **書くこと**: 対象 path、実行条件、上書き条件、承認が必要な操作。
  - **避けること**: 「必要に応じて実行する」のように対象と条件を曖昧にする。
- **選択規則**:
  - **対象**: 複数の file、section、template、script、候補案。
  - **書くこと**: 優先順位、tie-breaker、選ばない条件。
  - **避けること**: 候補選択を文脈依存の判断だけにする。
- **失敗時の挙動**:
  - **対象**: 入力不足、file 不在、format 不正、command 失敗、検証不能。
  - **書くこと**: retry、fallback、質問、中断、報告のどれを行うか。
  - **避けること**: 失敗後の処理を出力契約だけに押し込む。
- **完了条件**:
  - **対象**: 編集、生成、review、validation、報告。
  - **書くこと**: 何を確認できれば task 完了とみなすか。
  - **避けること**: 作業実行と完了判定を同一視する。

## 処理のスクリプト化

- 入力から出力を機械的に決められる反復処理は script 化する。
- ユーザー意図、設計判断、リスク判断が必要な処理は script 化の対象にしない。
- script への代替は、その契約によって別スコープ化されたものとして扱い、元の手順に同じ契約を重複させない。
  - 元の手順には script の参照と実行条件だけを残す。
