# Structured Artifact Rules

<!--
運用法:
- 別の文書でこの `structured_artifact_rule` を参照し、その文書に登場する名前へ文脈ごとの解釈を割り当てて使う。
- 例: プログラミング言語では、`スコープ` に class を、`名前` に variable name や method name などを割り当てる。
- この rule は文脈へ写像して使うため、スコープ、名前、責務、参照関係の厳密性を重視する。
-->

## 用途

* **目的**: **成果物** を簡潔で追跡可能な形に保ち、利用者と作成者が同じ **スコープ**、**名前**、**責務**、**参照関係** をたどれるようにする。

* **使用場面**: **成果物** の作成、内容変更、レビュー、リバイズで、冗長性、**スコープ** 関係、**名前** を確認する。対象を **書き換え** る場合は、**責務** の範囲と **参照関係** の **情報保持** を確認する。

* **対象**: **スコープ**、**名前**、**責務**、**参照関係** を持つ **成果物**。

* **位置づけ**: 特定の形式に依存しない抽象 rule として使う。形式固有の詳細規約は、必要な場合だけ別 rule、補助基準、構造定義に委ねる。

## 定義

`[*]`: 外部の interpretation file で文脈ごとの解釈を割り当てる定義語に付ける。
この節の定義語は、この rule 内で一貫して使う内部定義名である。`[*]` は、そのうち外部の interpretation file で直接割り当てる語だけに付ける。
**除外規則**: `[*]` の定義から導出できる定義には `[*]` を付けない。そうした語は、文書全体で使うなら `[*]` なしの内部定義名としてこの節に置き、局所的に使うだけなら使う最小 **スコープ** で説明する。

* `[*]` **成果物**: 作成、利用、確認、変更の対象になる構造化された出力。

* `[*]` **スコープ**: **成果物** 内で意味境界を作る単位。

* `[*]` **名前**: **成果物** 内の対象を指すための表現。

* `[*]` **責務**: **スコープ** が担う役割、制約、期待される働き。

* `[*]` **参照関係**: **成果物** 内外の対象間の関係。

* **書き換え**: **成果物** の内容、**スコープ**、**名前**、**責務**、**参照関係** を変更すること。

* **情報保持**: **書き換え** 後も必要な **責務**、**参照関係** を再現できるようにする確認。

## 原則

* **解釈の不確実性の減少**: **スコープ** の境界、**責務**、**参照関係** を明示し、同じ概念を同じ **名前** で扱う。

* **読解/利用コストの減少**: 利用順序と参照導線を構造で示し、必要な情報へ短く到達できるようにする。

* **構造選択**: 形式化を目的にせず、**スコープ** の **責務** と **参照関係** を最も短く安定して表せる構造を選ぶ。

* **適用軸**: **冗長性の排除**、**構造化**、**厳密化** を組み合わせて確認する。

* **書き換え時の保持確認**: 削除、統合、抽象化、移動、**名前** 変更の後も、必要な **責務** の範囲と **参照関係** を再現できるようにする。

## 適用軸

この節では、**成果物** を評価または作成するときの中心軸として、**冗長性**、**構造化**、**厳密化** を扱う。**情報保持** は同列の軸ではなく、対象を **書き換え** る場合に **書き換え時の確認** で扱う。

### 冗長性

**冗長性** は、削っても **成果物** の意味、**責務**、**参照関係** が変わらない重複や過剰な内容を指す。

* **削除基準**: 削っても **成果物** の意味、**責務** の範囲、**参照関係** が変わらない内容は削除候補にする。

* **削除してよいもの**:

  * 同じ **責務** または **参照関係** を別箇所で繰り返しているもの。

  * 作成過程の名残で、現在の **責務** や **参照関係** に寄与しないもの。

  * 上位 **スコープ** の **責務** から外れる詳細。

  * 独立した **責務** や **参照関係** を区別していない列挙。

  * 上位 **スコープ** や **名前** から読み取れる補助情報。

  * 移動、統合、抽象化の後に役割を失った構造。

* **保持すべきもの**:

  * 削ると **責務** の範囲を再現できないもの。

  * **責務** の範囲または **参照関係** を固定しているもの。

  * 同じ **名前** や構造に見えても、異なる **責務** や **参照関係** を区別しているもの。

* **圧縮の限界**: **責務** または **参照関係** を支える関係が不明になるなら残す。

### 構造化

**構造化** は、**スコープ** を分け、配置し、**名前**、**責務**、**参照関係** を追跡できる形にすることを指す。

* **スコープ分解**:

  * **スコープ** は独立した **責務** を担い、必要な **参照関係** を持つ。

  * 複数の役割を持つ **スコープ** は分離し、独立しない細片は統合する。

  * 大きな **スコープ** では、子 **スコープ** に共通する **責務** と参照導線を明示する。

* **スコープ分類**:

  * **根スコープ**: 対象作業で扱う最上位の **スコープ**。

  * **親スコープ**: 子 **スコープ** の関係と共通情報を受け持つ **スコープ**。

  * **葉スコープ**: 子 **スコープ** を持たない最小の意味単位。

* **スコープ関係**:

  * **スコープ** 間の **参照関係** は、独立関係、包含関係、依存関係に分けて扱う。

  * **独立関係**: **スコープ** 同士が別の **責務** を担い、一方の変更が他方の **責務** を変えない関係。

  * **包含関係**: 親 **スコープ** が子 **スコープ** を含み、共通する **責務** と参照導線を受け持つ関係。

  * **依存関係**: ある **スコープ** が別の **スコープ**、**名前**、**責務**、または **参照関係** に依存する関係。

  * 使用場面、作業手順、削除基準、検証観点から使われる **スコープ** には、到達できる参照導線を置く。

* **一貫性**:

  * 同じ **親スコープ** に属する並列 **スコープ** は、形式、**責務**、粒度、順序を揃える。

  * 同じ **責務** を持つ対象は、**名前**、境界、順序の揃え方を合わせる。

  * 差分が **責務** の違いなら、暗黙の例外にせず構造として示す。

* **構造選択**:

  * **名前** と **責務** を持つまとまりは、独立した **スコープ** にする。

  * 比較対象は、同じ軸で並べる。

  * 手順や依存順があるものは順序を示し、順序が **責務** に関係しないものは同列に扱う。

  * 構造を増やしても **責務** と **参照関係** が明確にならない場合は、自然言語で関係を示す。

### 厳密化

**厳密化** は、**名前**、定義位置、**責務**、**参照関係** の曖昧さや揺れを減らすことを指す。

* **名前**:

  * 多用する概念には名前を与え、同じ概念には同じ名前を使う。

  * 同じ名前を複数の概念に使わない。

  * **名前** は、役割、**責務**、分類、接続関係のうち区別に必要な情報を含める。

  * **スコープ** から回復できる **名前** は、**名前** 内で繰り返さない。

  * 定義と実値、目印と内容、状態と判定、規則と結果を混同させない。

* **名前分類**:

  * **内部定義名**: **成果物** 内で定義され、その **成果物** 内の参照単位になる **名前**。使用箇所を含む最小の **スコープ** に定義を置く。

  * **外部定義名**: **成果物** の外側の基準で定義され、**成果物** 内では原則として再定義しない **名前**。必要なら周辺説明で役割を補う。

* **定義位置**:

  * 定義は、その概念の使用箇所を含む最小の **スコープ** に置く。

  * 使用範囲より広すぎる場所に局所概念を置かない。

  * 使用箇所の一部から参照できない狭すぎる場所に定義を置かない。

* **表記統一**:

  * 短縮名、正式名、別名、表記揺れを混在させない。

  * 形式固有の **名前**、外部の名前体系は、参照元の表記に合わせる。

## 書き換え時の確認

この節は、対象を **書き換え** る作業でだけ使う。レビューのみでは、**情報保持** を **冗長性**、**構造化**、**厳密化** と同列の評価軸にしない。

### 情報保持

* **位置づけ**: **情報保持** は、**冗長性の排除**、**構造化**、**厳密化** と同列の評価軸ではなく、対象を **書き換え** るときの確認義務として扱う。

* **適用時**: 内容変更、リバイズ、削除、統合、抽象化、移動、**名前** 変更、表記統一、構造変更など、対象を **書き換え** るときに必ず確認する。

* **非適用時**: レビューのみでは評価軸にしない。新規作成では、将来の変更時に保持すべき **責務** を明確にする範囲に留める。

* **確認問い**: 変更後も、**根スコープ** から **葉スコープ** まで、必要な **責務** の範囲と **参照関係** を再現できるか。

* **保持対象**:

  * **責務** を固定する規則。

  * **名前** と **参照関係** の対応。

  * **参照関係** と参照導線。

* **保持方法**:

  * 詳細を削る場合は、**責務** を保つ上位規則または参照導線として残す。

  * 統合する場合は、統合前の差分が **責務** の違いか単なる表記差かを確認する。

  * **名前** 変更する場合は、参照元、参照先、**外部定義名** との対応を失わせない。

  * 移動する場合は、利用元から移動先へ到達できる導線を残す。

## 作業手順

この節では、作業種別を分類し、最小手順に従って適用軸と **書き換え** 時確認を適用する。

### 作業種別

* **新規作成**: **根スコープ** の **責務** と初期 **スコープ** 関係を定めて作る。

* **内容変更**: 既存 **成果物** の意味、**責務**、**参照関係** を追加、削除、置換、更新する。

* **レビュー**: **冗長性**、**構造化**、**厳密化** の問題と修正候補を特定し、対象を **書き換え** ない。

* **リバイズ**: レビューと同じ確認を行い、意味や **責務** を変えない範囲で対象を **書き換え** る。

### 最小手順

1. **入力確認**: 作業種別、**根スコープ** の **責務**、編集可否を確認する。不足が **成果物** の **責務** に影響する場合は、作業前に確認する。
2. **スコープ把握**: **根スコープ**、**親スコープ**、**葉スコープ** を特定し、**責務** と **参照関係** をたどる。
3. **変更方針**: 新規作成では初期 **スコープ** 関係を決める。内容変更では変更先 **スコープ**、新規 **スコープ** の要否、抵触、影響範囲を確認する。レビューとリバイズでは優先対象の **スコープ** を決める。
4. **適用**: **冗長性**、**構造化**、**厳密化** の順に確認し、必要な作成、変更、指摘化を行う。
5. **書き換え確認**: 対象を **書き換え** た場合は、変更した **スコープ**、その **親スコープ**、参照先、参照元を確認する。**責務**、**名前**、**参照関係** が外側へ影響する場合は確認範囲を広げ、**情報保持** を確認する。
6. **出力**: 変更箇所または指摘、未確定事項、実行した検証を報告する。対象を **書き換え** た場合は、**情報保持** 確認の結果も報告する。

## 出力/確認観点

### 出力

* **レビューのみ**: 指摘、最小修正案、未確定事項を返す。

* **書き換えあり**: 変更箇所、変更内容、**情報保持** 確認の結果、未確定事項を返す。

* **検証あり**: 実行した確認操作と結果を返す。

### 確認観点

* **冗長性の排除**: 削っても **責務** や **参照関係** が変わらない重複を削り、必要な差分は残したか。

* **構造化**: **スコープ**、**責務**、**参照関係** が追えるか。

* **厳密化**: 同じ概念に同じ **名前** を使い、定義位置と表記が安定しているか。

* **書き換えありの場合**: 移動、抽象化、統合、削除、**名前** 変更で **責務** と **参照関係** を失っていないか。
