# Writing Rules

## 用途

- **指示文書** を lean に保つための詳細規則。
- **指示文書** の作成、更新、レビューで、冗長性、構造、用語・表記・挙動の厳密化、情報保持の判断が必要な場合に読む。
- **指示文書**: `SKILL.md`、bundled resources、プロンプト本文など、Codex の動作を規定する文書。

## 目次

- **Notation**
- **冗長性の排除**
- **構造化**
- **情報保持**
- **レビュー検査への適用**
- **厳密化**

## Notation

- **description item**: `**ラベル**: 内容` の形で、対象、条件、例などの属性を示す list item。
- **field**: description item または record の中で使う属性名。
- **field set**: 同じ対象群に対して使う field の集合。
- **field order**: 同じ field set の field を並べる順序。
- **unordered description**: description item を unordered list として並べたもの。
- **ordered description**: description item を ordered list として並べたもの。
- **record**: 複数対象を同じ **field set** で扱うとき、各対象を親 list item とし、その child list に `**field**: value` 形式の description item を並べた構造。
  - **例**:
    - **target1**:
      - **field A**: value A
      - **field B**: value B
    - **target2**:
      - **field A**: value A
      - **field B**: value B

## 冗長性の排除

- 下記の **書く内容** と **書く目的** のどれにも該当しない指示は削除候補にする。
- 読みにくい細分化、機械的すぎる記法、Markdown の乱用は避ける。
- 同じ指示を異なる箇所に何度も書かない。
- 重複がある場合は、片方を削るか、抽象原則と具体規則に分ける。
- 要約と詳細の重複は許可する。詳細から要約を回復できる場合でも、常時参照する判断軸として必要なら残してよい。
- ただし、要約には具体手順、例、ラベル、個別条件、可変値を入れない。

### 削除候補条件

- 既存の上位原則、workflow、出力契約から容易に推論できる。
- Skillの修正過程でのレガシー。
- 同じ判断条件、禁止事項、手順、例が別の場所にある。
- 特定タスクの一回限りの値を、再利用可能な規則として書いている。
- **指示文書** の入力、手順、出力、検証条件のいずれにも影響しない。
- 例や補足を削っても、同じ判断を **構造化**、**情報保持**、**厳密化** の規則から再現できる。

### 書く内容

- **指示文書** として記述するのは、基本的に入力、手順、出力の指定。
  - **入力**: 参照する情報、受け取る値、前提、不明情報の扱い。
  - **手順**: 調査、判断、変換、編集、レビューなどの進め方。
  - **出力**: 形式、粒度、保存先、返答内容、schema、検証条件。

### 書く目的

- **指示文書** を記述する目的は、動作の不確実性の減少と、推論コストの減少。
  - **動作の不確実性の減少**: 判断基準、責務境界、禁止事項、出力契約の明示。
  - **推論コストの減少**: 読むべき情報、使うべき tool、参照すべき file、確認すべき条件の明示。

## 構造化

- 指示は Markdown を使って構造化する。
- 構造化の基本は **description item** である。
- 判断条件、属性、比較対象を列挙する場合は、**構造の一貫性** に反しない範囲で **description item** を優先する。
- 下記は **description item** の代表的な型である。
  - **表記記号**: `NP` は名詞句、`AdjP` は形容詞句、`VP` は動詞句、`PP` は前置詞句、`CP` は条件節、`XP` は任意の句、`LHS` は label、`RHS` は内容を表す。
  - **同格型**: `NP : NP` : `LHS` is `RHS`
  - **述語型**: `NP : AdjP / VP / Clause` : `LHS` has / does / satisfies `RHS`
  - **列挙型**: `NP : List[XP]` : `LHS` consists of `RHS1`, `RHS2`, ...
  - **文脈型**: `PP : Clause` : In / under / for `LHS`, `RHS`
  - **条件型**: `CP : Clause` : If / when `LHS`, then `RHS`
  - **フレーム型（record）**: `NP : List[NP : XP]` : `LHS` = { `field1` = `RHS1`, `field2` = `RHS2`, ... }

### 分解とモジュール化

- **モジュール**: 指示文書の中で、独立した責務、判断、契約のいずれかを担う記述上の構成要素。
- 大きな指示を、意味のあるモジュールに分ける。
- 各モジュールでは責務、判断、契約を明記する。

### 分離の種類

- **独立な分離**: 互いに前後関係や参照関係を持たない関心を分ける。
- **依存のある分離**: モジュールどうしの前後関係、参照関係、前提関係を明示する。

### モジュールの一般化

- モジュールには、特定タスクの値をハードコードしない。
- 会社名、日付、path、score、field 値などの可変値は、入力、設定、template、schema、例に分離する。
- モジュール本文には、具体例ではなく再利用できる判断構造を書く。

### Markdown による構造化

- **section**:
  - **解釈**: 名前を持つ大きなまとまり。
  - **形式**: `Block[Title, Body]`
  - **分離の種類**: 独立な分離、依存のある分離。
  - **用途**: 独立した責務、関心、workflow phase、または順序や参照関係を持つまとまりを分ける。
  - **避ける例**: 責務境界のない細かい section を増やす。
  - **例**:
    ```markdown
    ## 調査

    対象、根拠、未確認事項をまとめる。
    ```

- **unordered list**:
  - **解釈**: 順序を持たない同型 item の集合。
  - **形式**: `Set[Item]`
  - **分離の種類**: 独立な分離。
  - **用途**: 同じ型の項目、選択肢、チェックリスト、分類を並べる。
  - **避ける例**: 原則、例外、補足、例示を同じ description を使わない unordered list に混ぜる。
  - **例**:
    ```markdown
    - 調査
    - 判断
    - 編集
    - 検証
    ```

- **unordered description**:
  - **解釈**: 1 つの対象に対する field の集合。
  - **形式**: `Record[Field : Value]`
  - **分離の種類**: 独立な分離。
  - **用途**: 1 つの対象に対する属性を並べる。
  - **避ける例**: 属性名を付けずに異なる役割の説明を並べる。
  - **例**:
    ```markdown
    - **対象**: ordered list
    - **用途**: 手順を並べる。
    - **避ける例**: 順序のない分類に使う。
    ```

- **ordered list**:
  - **解釈**: 順序または依存関係を持つ item の列。
  - **形式**: `Seq[Step]`
  - **分離の種類**: 依存のある分離。
  - **用途**: 時系列、実行手順、優先順位、評価段階を並べる。
  - **避ける例**: 依存関係が全順序でない場合に ordered list を使う。
  - **例**:
    ```markdown
    1. 調査する。
    2. 判断する。
    3. 編集する。
    4. 検証する。
    ```

- **ordered description**:
  - **解釈**: 順序を持つ各 step に record を付けたもの。
  - **形式**: `Seq[Step : Record]`
  - **分離の種類**: 依存のある分離。
  - **用途**: 順序を持つ各 step に、条件、入力、出力を付ける。
  - **避ける例**: step 間の順序がない属性説明に ordered description を使う。
  - **例**:
    ```markdown
    1. **調査**:
       - **入力**: 対象文書
       - **出力**: 根拠一覧

    2. **判断**:
       - **入力**: 根拠一覧
       - **出力**: 修正方針
    ```

- **child list**:
  - **解釈**: 上位 item に従属する下位 item を持つ構造。
  - **形式**: `Tree[Parent, Child]`
  - **分離の種類**: 依存のある分離。
  - **用途**: 上位 item に従属する説明、条件、補足、例を置く。
  - **避ける例**: 上位 item と関係しない内容を child list に置く。
  - **例**:
    ```markdown
    - **規則**: ordered list は依存関係がある場合に使う。
      - **条件**: 前の step が後の step の前提になる。
      - **避ける例**: 独立した分類を ordered list にする。
    ```

- **record**:
  - **解釈**: 複数対象を同じ field set で列挙または比較する構造。
  - **形式**: `Seq[Subject : Record[Field : Value]]`
  - **分離の種類**: 独立な分離、依存のある分離。
  - **用途**: 同じ種類の対象、同じ判断、同じ比較軸を並べる。
  - **避ける例**: 1 つの対象の属性だけを書く場合や、長い文章であることだけを理由に record 化する。
  - **例**:
    ```markdown
    - **target1**:
      - **field A**: value A
      - **field B**: value B
    - **target2**:
      - **field A**: value A
      - **field B**: value B
    ```

### 構造の一貫性

- **対象**: 同じ種類の対象、同じ判断、同じ比較軸、同じ workflow step、同じ入出力契約を複数回書く箇所。
- **基本規則**: 同じ意味のまとまりは、同じ Markdown 構造、同じ見出し粒度、同じ item order、同じ label 名で書く。
- **section の一貫性**:
  - 同じ workflow phase や同じ責務を扱う section は、同じ見出し粒度と同じ内部構造で書く。
  - ある section だけ目的、入力、出力、検証条件などの項目が欠ける場合は、省略理由が判断に影響しないか確認する。
- **list の一貫性**:
  - 同じ種類の項目を並べる場合は、unordered list、ordered list、description のどれを使うかを揃える。
  - 順序や依存関係がある項目だけ ordered list にする。
  - 並列項目では、各 item の文法、粒度、終止形を揃える。
- **description の一貫性**:
  - 同じ役割の属性には同じ label 名を使う。
  - 同じ label は同じ意味で使い、異なる意味には別の label を使う。
  - 類似した description では、label の順序を固定する。
- **record の一貫性**:
  - 複数対象を同じ field set で列挙または比較する場合は **record** を使う。
  - 同じ判断には同じ field set と同じ field order を使う。
  - 一部の対象だけに必要な field は、任意 field として扱うか、対象群を分ける。
  - 1 つの対象の属性だけを書く場合は **unordered description** を使う。
  - 長い文章であることだけを理由に **record** 化しない。
- **child list の一貫性**:
  - child list に置く情報の役割を、条件、補足、例、避ける例などで揃える。
  - 同じ親 item の下で、互いに異なる役割の child item を無標識に混ぜない。
- **分割条件**: 対象ごとに必要な構造、判断順序、出力契約が変わる場合は、同じ構造に押し込まず、別の **section**、別の **record**、または別の list に分ける。
- **確認問い**: 同じ種類の対象が同じ形で読めるか。ある対象だけ項目が増減していないか。同じ label や field 名が別の意味で使われていないか。比較対象を横断して同じ順序で読めるか。

## 情報保持

- 削除、統合、抽象化の前に、残すべき入力、判断条件、出力契約を特定する。
- 具体例を削る場合は、同じ判断を再現できる抽象規則を残す。
- 名称を変更する場合は、変更前後で参照先と意味境界が変わらないことを確認する。
- 編集後に、削除した内容が別の **section**、**record**、schema、template から回復できるか確認する。

### 回復可能性

- **回復可能**: 削除または統合した内容と同じ判断を、残った入力、手順、出力契約、reference、schema、template から再現できる。
- **回復不能**: 削除または統合によって、必要な入力、判断条件、禁止事項、検証条件、出力形式のいずれかが再現できない。
- 回復不能な内容は削除しない。短縮する場合は、抽象規則、**record** field、schema、template、または reference への明示参照として残す。

## レビュー検査への適用

- **冗長性検査**:
  - **確認問い**: その記述は入力、手順、判断、出力、検証のいずれかを変えるか。別の記述から同じ粒度で回復できないか。
  - **修正方針**: 不要なら削除し、必要なら要約と詳細、抽象原則と具体規則に分ける。
- **構造化検査**:
  - **確認問い**: 見出し、list、手順、補助資料の分担は内容の責務と一致するか。読む順序、分岐、停止条件が誤解なく追えるか。
  - **修正方針**: 責務を分離し、一元化し、同じ種類の対象は揃った構造で表す。
- **厳密化検査**:
  - **確認問い**: 名称、表記、参照、入力契約、副作用、選択規則、失敗時の挙動、完了条件に曖昧さや揺れがないか。
  - **修正方針**: 意味境界を明確にし、同じ意味には同じ名称、異なる意味には異なる名称を使う。
- **情報保持検査**:
  - **確認問い**: 削除、統合、抽象化、移動の後も、必要な入力、判断、禁止事項、出力、検証を再現できるか。
  - **修正方針**: 具体例や詳細を削る場合は、同じ判断を再現できる抽象規則、契約、schema、template、または参照案内を残す。

## 厳密化

### 用語の厳密化
- 指示文書内で多用する概念には用語を割り当てる。
- 用語はプログラミング言語における変数や参照情報のように扱う。
  - 同じ用語に複数の意味を持たせない。
    - 抽象度の高い語を複数の概念に使い回さない。
    - 文脈ごとに意味が変わる用語にしない。
  - 同じ意味に複数の用語を持たせない。
    - 短縮形と正式名称を混ぜない。
    - 同一概念内で英日表記を混ぜない。
    - 外部定義語、Markdown 構造名、Codex 固有語は英語表記のまま維持してよい。

- 用語は外部定義用語と内部定義用語に分けられる。

- **外部定義用語**: 指示文書外のツールやドキュメントで定義された名称。
  - **対象**: code、path、field name、tool name、schema key、API 名、command、英語 documentation の terminology。
  - **表記**: 外部定義用語は code span で示す。
    - **例**: `README.md`
    - **例**: `name`
    - **例**: `git diff`
- **内部定義用語**: 指示文書内で定義された名称。
  - **対象**: section名、概念名、責務名、判断基準名、workflow 名、step 名、入力名、出力名、schema 上の意味名、review 観点名、scoring 観点名。
  - **表記**: 内部定義用語は bold で示す。
    - **例**: **モジュール**
    - **例**: **関心分離**
    - **例**: **出力契約**

#### 用語の定義方法

- 文書内では、作業判断に影響する用語を使う前に、その定義と解釈を明示する。
- この規則は外部定義用語と内部定義用語の両方に適用する。
  - 外部定義用語は外部名を変更せず、code span で示す。
  - 外部名だけでは型や役割が曖昧な場合は、周辺の説明名や field 名で補う。
- 名称または説明名にはその（型 / 意味）に加え、必要なら接続関係、定義と実値、目印と内容の区別を含める。
- **外部定義用語の例**:
  - `placeholder` token: token / 後で値を差し込むための外部定義 marker。
  - `name` field key: schema key / 名称を保持する外部定義 field key。
  - `--output` CLI option: option / 出力先を指定する外部定義 command line option。
  - `git diff` command: command / 比較対象間の変更差分を出力する外部定義 command。
- **内部定義用語の例**:
  - **input slot definition**: definition / 入力値を受け取る枠の定義。
  - **input value instance**: value / **input slot definition** に渡される実際の値。
  - **comment anchor marker**: marker / コメントを付ける位置を示す目印。
  - **comment body text**: text / **comment anchor marker** に紐づけて表示する本文。
- 型を省いた名称は、受け口と差し込むもの、定義と実値、目印と内容、規則と結果を混同させる。
  - **悪い例**: **input**: 入力枠、入力値、入力 file、入力条件のどれか分からない。
  - **悪い例**: **placeholder**: 置換される token なのか、差し込む値なのか分からない。
  - **悪い例**: **comment**: コメント位置なのか、コメント本文なのか分からない。
  - **悪い例**: **field**: schema 上の field 定義なのか、実際の field 値なのか分からない。

#### 構造と用語
- **スコープ構造**: 意味境界を分けるような**section**、**record**、schema、template、directory
  - **例**: `テストデータ集合` (`テスト`はスコープ、`データ集合`はその中の用語)
  - file pathもdirectoryによるスコープ構造を含む。
- スコープが明確な時、内部、外部の用語から、スコープ由来の単語を積極的に省く。
  - **例**: `テスト`セクションでは単に`テストデータ集合`を`データ集合`と呼ぶ。
  - 特にpathは短くなるように徹底する。
  - 用語だけで意味境界を分けない。
- スコープ名で同じ情報を重複させない。
- スコープの内容に対し、スコープの名前が適切かどうかを常に考慮する。

### プロンプトの挙動の厳密化

- 同じ入力、同じ状態、同じ参照情報では、同じ判断と同じ副作用になるように書く。
- 特に動作するシステムに干渉させる指示は一層厳密化する。

- **入力契約**:
  - **対象**: 必須入力、任意入力、既定値、不明情報。
  - **書くこと**: 何を受け取り、何がない場合に質問、仮定、中断のどれを選ぶか。
  - **避けること**: 入力不足時の扱いを workflow の推測に任せる。
- **副作用契約**:
  - **対象**: ファイル読み書き、生成、削除、command 実行、sub-agent 起動。
  - **書くこと**: 対象 path、実行条件、上書き条件、承認が必要な操作。
  - **避けること**: 「必要に応じて実行する」のように対象と条件を曖昧にする。
- **選択規則**:
  - **対象**: 複数の file、section、template、script、候補案。
  - **書くこと**: 優先順位、tie-breaker、選ばない条件。
  - **避けること**: 候補選択を文脈依存の判断だけにする。
- **失敗時の挙動**:
  - **対象**: 入力不足、file 不在、format 不正、command 失敗、検証不能。
  - **書くこと**: retry、fallback、質問、中断、報告のどれを行うか。
  - **避けること**: 失敗後の処理を出力契約だけに押し込む。
- **完了条件**:
  - **対象**: 編集、生成、review、validation、報告。
  - **書くこと**: 何を確認できれば task 完了とみなすか。
  - **避けること**: 作業実行と完了判定を同一視する。
