# Skill Writer

## 対象と成果物

- **新規作成**:
  - **対象**: 新しい skill。
  - **成果物**: 目的、適用対象、入力、手順、出力、副作用条件、禁止事項、検証条件を持つ skill。
- **内容追加**:
  - **対象**: 既存 skill。
  - **成果物**: 既存構造に沿って追加内容を反映した skill。
- **既存 skill リバイズ**:
  - **対象**: 既存 skill。
  - **成果物**: 既存の目的、入力、手順、出力、副作用条件、禁止事項、検証条件を保持しつつ、冗長性、構造、命名、情報保持を改善した skill。
- **既存 skill レビュー**:
  - **対象**: 既存 skill。
  - **成果物**: 冗長性、構造の混線、命名の曖昧さ、task-specific な混入、情報落ち、副作用条件の抜けの findings。

## 中核原則

- 実際の挙動に影響しない箇所についても重視する。
- 人間と LLM にとって読みやすく、理解しやすい skill を目指す。
  - **冗長性の排除**: 推測可能な説明や重複指示を避ける。
  - **構造化**: 責務、判断、出力契約を文書構造で分ける。
  - **厳密化**: 名称、表記、スコープの曖昧さを減らす。
  - **情報保持**: 削除、統合、抽象化によって入力情報、判断条件、出力契約、副作用条件を失わない。

## 参照

- **`skill-creator`**: 新規作成時の scaffold、template、validation、skill folder の基本構成を確認する場合に参照する。
- **常時参照**: skill-writer を使うときは、作業前に `skills/common/structured_document_rule.md` と `skills/common/instruction_rule.md` を必ず読む。
  - **`skills/common/structured_document_rule.md`**: 冗長性、構造、用語・表記、情報保持の詳細判断の前提として読む。
  - **`skills/common/instruction_rule.md`**: 指示文書の入力契約、副作用契約、選択規則、失敗時の挙動、完了条件を確認する前提として読む。

## ワークフロー

### 新規作成

1. **要件確認** で目的、使用場面、対象読者または実行者、入力、出力、副作用条件、禁止事項を固定する。
2. **中核原則** と **参照** に照らして、ユーザーの指示を skill の入力、手順、出力、副作用条件へ整理する。
3. ユーザーの明示的な承認または続行指示を得たら、`skill-creator` を使って skill template を適用し作成する。
4. 確認済みの入力、手順、出力、副作用条件、禁止事項、検証条件が作成後の skill に残っているか確認する。
5. 完成した skill を **レビュー手順** で確認する。

### 内容追加

1. 対象 skill と追加内容を受け取る。内容が不足している場合は、**要件確認** を行う。
2. **中核原則** と **参照** に照らして、追加先の **section**、**record**、または手順を決める。
3. 既存 skill ファイルを編集する場合は、編集方針と副作用を提示し、ユーザーの承認または明示的な続行指示後に反映する。
4. 既存 skill の構造と表記に合わせて追加内容を反映する。
5. 変更した skill を **レビュー手順** で確認する。

### 既存 skill リバイズ

1. 対象 skill folder または指定された file 群と、リバイズ目的を受け取る。リバイズ目的が不足している場合は、**要件確認** を行う。
2. **対象範囲確認** と **参照確認** を行い、対象 skill の目的、入力、手順、出力、副作用条件、禁止事項、検証条件を固定する。
3. `skills/common/structured_document_rule.md` の **共通: スコープ単位確認** と `skills/common/instruction_rule.md` の **レビューとリバイズの反復** に従い、対象 skill を実際に修正する。
4. **確認手順** の中で修正が行われた場合は、もう一度 **確認手順** を繰り返す。
5. この skill と参照 rule だけで判断できない問題は処理を停止し、ユーザーに判断を求める。
6. 修正箇所がなくなったら、可能な限り `skills/.system/skill-creator/scripts/quick_validate.py <path/to/skill-folder>` を実行し、`git diff` などで差分を確認する。

### 既存 skill レビュー

1. 対象 skill folder または指定された file 群を受け取る。
2. `SKILL.md` とレビュー対象の bundled resources の整合性を含めて、対象 skill を **レビュー手順** で確認する。

## 禁止事項

- ユーザーから与えられていない事実、要件、制約を確定情報として書かない。合理的な導出は仮案として示す。

## 汎用手順

### `SKILL.md` と bundled resources の役割分担

- **`SKILL.md` に置く候補**:
  - **対象と成果物**: skill が扱う対象と返す成果物。
  - **中核原則**: 常時参照する短い判断軸。具体手順、例、ラベル、個別条件を含めない。
  - **参照案内**: どの bundled resource をいつ読むか。
  - **ワークフロー**: `SKILL.md` で常時必要な実行順序、分岐条件、停止条件。詳細手順を bundled resources に移す場合は、参照案内だけを残す。
  - **出力契約**: ユーザーに返す形式、粒度、検証報告。
- **bundled resources に置くもの**:
  - **詳細規則**: 構造、用語・表記・挙動、情報保持、削除候補の判断条件。
  - **長い例**: `SKILL.md` に常時置く必要がない例。
  - **比較表や schema**: 必要なときだけ読む判断材料。
- **`agents/openai.yaml` に置くもの**:
  - **UI metadata**: skill list や chip に表示する名称、短い説明、default prompt。
  - **同期条件**: `SKILL.md` の目的、使用場面、trigger 文言を変更した場合に内容が古くないか確認する。
- **分離規則**: 同一粒度の同じ規則を重複させない。`SKILL.md` には要約、適用条件、満たすべき契約を置き、細かい判断方法は必要に応じて bundled resources に分ける。

### 要件確認

- **確認項目**:
  - **目的**: skill が解決する問題。
  - **使用場面**: skill を使うタイミング。
  - **skill 名**: frontmatter `name` と呼び出し時の `$skill-name`。
  - **trigger 文言**: frontmatter `description` に入れる使用条件。
  - **作成先 path**: skill folder の保存先。
  - **対象読者または実行者**: skill を読む、または実行する主体。
  - **入力**: ユーザーから受け取る値、参照する file、前提。
  - **出力**: 返答形式、編集対象、保存先、検証条件。
  - **副作用条件**: ファイル編集、生成、削除、command 実行、sub-agent 起動などの実行条件。
  - **bundled resources**: 必要な `scripts/`、`references/`、`assets/`。
  - **禁止事項**: skill が行ってはいけないこと。
- **確認方法**:
  - 既存情報から合理的に決められる場合は、仮案を作ってユーザーに確認する。
  - skill の目的、使用場面、出力契約が不明な場合は、作成前に質問する。

### レビュー手順

1. **対象範囲確認**: 対象 skill folder、`SKILL.md`、レビュー対象の bundled resources を確認する。
2. **参照確認**: `skills/common/structured_document_rule.md` と `skills/common/instruction_rule.md` を読み、確認観点を固定する。
3. **検査**: **中核原則**、**参照**、下記の **レビュー検査** に照らして確認する。
4. **契約確認**: `SKILL.md` と bundled resources の契約を確認する。特に `references/`、`scripts/`、`agents/openai.yaml` の役割、script docstring、CLI、入出力契約が本文と一致するか見る。
5. **finding / 変更箇所分類**:
   - **確認が必要**: 設計方針、ユーザー意図、作業スコープの判断が必要な finding。修正へ進めずユーザーに確認する。
   - **局所修正候補**: ユーザーの指示範囲内で、意味と契約を変えずに直せる finding。レビューのみの依頼では編集せず、最小修正案として出す。
   - **出力カテゴリ**: **冗長性の排除**、**構造化**、**厳密化**、**その他の内容** のいずれかに分類する。
   - **重要度**: レビュー finding とリバイズ変更箇所には `high` または `low` を付ける。入力、判断、出力契約、副作用条件、情報保持に影響するものは `high`、局所的な読みやすさや表記改善は `low` とする。
6. **レビュー出力**: レビューのみの依頼では、**出力契約** の **レビューのみ** に従って findings を返す。
7. **編集後検証**: 新規作成または更新後は、可能な限り `skills/.system/skill-creator/scripts/quick_validate.py <path/to/skill-folder>` を実行し、未実行の場合は理由を報告する。既存 skill を編集した場合は、`git diff` などで差分を確認する。
8. **保存**: レビュー結果を保存する場合は `document-workflow` skill に基づく。

#### レビュー検査

- 常時参照する短いチェックリストとして使う。詳細な確認問いと修正方針は `skills/common/structured_document_rule.md` の **レビューとリバイズへの適用** と `skills/common/instruction_rule.md` の **指示文書固有の確認** に従う。
- **冗長性検査**: 入力、手順、判断、出力、検証、副作用に影響しない重複や推論可能な記述を確認する。
- **構造化検査**: 見出し、list、手順、補助資料の責務、読む順序、兄弟スコープ間の役割重複を確認する。
- **厳密化検査**: 名称、表記、参照、入力契約、副作用、選択規則、失敗時の挙動、完了条件を確認する。
- **情報保持検査**: 削除、統合、抽象化、移動の後も必要な入力、判断、禁止事項、出力、検証、副作用条件を再現できるか確認する。

## 出力契約

- **レビューのみ**:
  - **出力構造**: **冗長性の排除**、**構造化**、**厳密化**、**その他の内容** の 4 項目で出力する。
  - **finding**: 対象 file path、行番号、重要度 (`high` / `low`)、問題、最小修正案を含める。
  - **空カテゴリ**: 該当 finding がない場合は `なし` と書く。
- **リバイズあり**:
  - **出力構造**: **冗長性の排除**、**構造化**、**厳密化**、**その他の内容** の 4 項目で出力する。
  - **変更箇所**: 対象 file path、行番号、重要度 (`high` / `low`)、変更したスコープ、変更内容、情報保持確認の結果を含める。
  - **空カテゴリ**: 該当する変更箇所がない場合は `なし` と書く。
  - **未対応事項**: 4 項目の後に、判断保留、未検証、ユーザー確認待ちがある場合だけ書く。
- **編集あり**: **編集報告** として変更 file、変更要約、未対応事項を含める。
- **validation 実行時**: **validation 報告** として実行した command、結果、未実行の場合の理由を含める。
