# Skill Writer

## 対象と成果物

- **新規作成**:
  - **対象**: 新しい skill。
  - **成果物**: 目的、適用対象、入力、手順、出力、検証条件を持つ skill。
- **内容追加**:
  - **対象**: 既存 skill。
  - **成果物**: 既存構造に沿って追加内容を反映した skill。
- **既存 skill レビュー**:
  - **対象**: 既存 skill。
  - **成果物**: 冗長性、構造の混線、命名の曖昧さ、task-specific な混入、情報落ちの findings。

## 中核原則

- **冗長性の排除**: 推測可能な説明や重複指示を避ける。
- **構造化**: 責務、判断、出力契約を Markdown 構造で分ける。
- **厳密化**: 名称、表記、スコープの曖昧さを減らす。
- **情報保持**: 削除、統合、抽象化によって入力情報、判断条件、出力契約を失わない。

## 参照

- **`skill-creator`**: 新規作成、内容追加、既存 skill レビューで参照する。skill folder、`SKILL.md`、`references/`、`agents/openai.yaml` の役割分担もここに従う。
- **`references/writing_rules.md`**: 冗長性、構造、用語・表記・挙動の厳密化、情報保持の詳細判断に使う。特に大きな編集、構造変更、抽象化、レビューでは読む。

## ワークフロー

### 新規作成

1. **要件確認** で目的、使用場面、対象読者または実行者、入力、出力、禁止事項を固定する。
2. **中核原則** と **参照** に照らして、ユーザーの指示を skill の入力、手順、出力へ整理する。
3. ユーザーの明示的な承認または続行指示を得たら、`skill-creator` を使って skill template を適用し作成する。
4. 確認済みの入力、手順、出力、禁止事項、検証条件が作成後の skill に残っているか確認する。
5. 完成した skill を **レビュー手順** で確認する。

### 内容追加

1. 対象 skill と追加内容を受け取る。
2. **中核原則** と **参照** に照らして、追加先の **section**、**record**、または手順を決める。
3. 既存 skill ファイルを編集する場合は、編集方針と副作用を提示し、ユーザーの承認または明示的な続行指示後に反映する。
4. 既存 skill の構造と表記に合わせて追加内容を反映する。
5. 変更した skill を **レビュー手順** で確認する。

### 既存 skill レビュー

1. 対象 skill folder または指定された file 群を受け取る。
2. `SKILL.md` とレビュー対象の bundled resources の整合性を含めて、対象 skill を **レビュー手順** で確認する。

## 禁止事項

- ユーザーから与えられていない事実、要件、制約を確定情報として書かない。合理的な導出は仮案として示す。

## 汎用手順

### `SKILL.md` と bundled resources の役割分担

- **`SKILL.md` に置く候補**:
  - **対象と成果物**: skill が扱う対象と返す成果物。
  - **中核原則**: 常時参照する短い判断軸。具体手順、例、ラベル、個別条件を含めない。
  - **参照案内**: どの bundled resource をいつ読むか。
  - **ワークフロー**: `SKILL.md` で常時必要な実行順序、分岐条件、停止条件。詳細手順を bundled resources に移す場合は、参照案内だけを残す。
  - **出力契約**: ユーザーに返す形式、粒度、検証報告。
- **bundled resources に置くもの**:
  - **詳細規則**: 構造、用語・表記・挙動、情報保持、削除候補の判断条件。
  - **長い例**: `SKILL.md` に常時置く必要がない例。
  - **比較表や schema**: 必要なときだけ読む判断材料。
- **分離規則**: 同一粒度の同じ規則を重複させない。`SKILL.md` には要約、適用条件、満たすべき契約を置き、細かい判断方法は必要に応じて bundled resources に分ける。

### 要件確認

- **確認項目**:
  - **目的**: skill が解決する問題。
  - **使用場面**: skill を使うタイミング。
  - **skill 名**: frontmatter `name` と呼び出し時の `$skill-name`。
  - **trigger 文言**: frontmatter `description` に入れる使用条件。
  - **作成先 path**: skill folder の保存先。
  - **対象読者または実行者**: skill を読む、または実行する主体。
  - **入力**: ユーザーから受け取る値、参照する file、前提。
  - **出力**: 返答形式、編集対象、保存先、検証条件。
  - **bundled resources**: 必要な `scripts/`、`references/`、`assets/`。
  - **禁止事項**: skill が行ってはいけないこと。
- **確認方法**:
  - 既存情報から合理的に決められる場合は、仮案を作ってユーザーに確認する。
  - skill の目的、使用場面、出力契約が不明な場合は、作成前に質問する。

### レビュー手順

- **中核原則** と **参照** に照らして確認する。
- 設計方針、ユーザー意図、作業スコープの判断が必要な finding は、修正へ進めずユーザーに確認する。
- `SKILL.md` と bundled resources の契約を確認する。特に `references/`、`scripts/`、`agents/openai.yaml` の役割、script docstring、CLI、入出力契約が本文と一致するか見る。
- 新規作成または更新後は、可能な限り `skill-creator` skill folder 配下の `scripts/quick_validate.py <path/to/skill-folder>` を実行し、未実行の場合は理由を報告する。
- 既存 skill を編集した場合は、`git diff` などで差分を確認する。
- **レビュー検査** は 下記の **レビュー検査** に従う。
- レビュー結果を保存する場合は `document-workflow` skill に基づく。
- レビュー結果の内、ユーザーの指示を経ずに改善できるものを提案する。

#### レビュー検査

- **冗長性検査**:
  - **確認問い**: その記述は入力、手順、判断、出力、検証のいずれかを変えるか。別の記述から同じ粒度で回復できないか。
  - **修正方針**: 不要なら削除し、必要なら要約と詳細、抽象原則と具体規則に分ける。
- **構造化検査**:
  - **確認問い**: 見出し、list、手順、補助資料の分担は内容の責務と一致するか。読む順序、分岐、停止条件が誤解なく追えるか。
  - **修正方針**: 責務を分離し、一元化し、同じ種類の対象は揃った構造で表す。
- **厳密化検査**:
  - **確認問い**: 名称、表記、参照、入力契約、副作用、選択規則、失敗時の挙動、完了条件に曖昧さや揺れがないか。
  - **修正方針**: 意味境界を明確にし、同じ意味には同じ名称、異なる意味には異なる名称を使う。
- **情報保持検査**:
  - **確認問い**: 削除、統合、抽象化、移動の後も、必要な入力、判断、禁止事項、出力、検証を再現できるか。
  - **修正方針**: 具体例や詳細を削る場合は、同じ判断を再現できる抽象規則、契約、schema、template、または参照案内を残す。

## 出力契約

- **レビューのみ**: **findings** として対象 file path、行番号、該当する原則、問題、最小修正案を含める。
- **編集あり**: **編集報告** として変更 file、変更要約、未対応事項を含める。
- **validation 実行時**: **validation 報告** として実行した command、結果、未実行の場合の理由を含める。
