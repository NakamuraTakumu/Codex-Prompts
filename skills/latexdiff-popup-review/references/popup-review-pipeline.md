# Popup Review Pipeline

## Purpose

This reference records the current batch pipeline for popup-comment review copies derived from latexdiff output.

The pipeline starts after a raw latexdiff artifact already exists. That raw artifact should be generated by the `latexdiff` skill first rather than by an ad hoc manual substitute.

The review pass is defined by the existence of the diff artifact, not by who made the original edit. The same pipeline applies whether the authoring side was Codex or someone else.

## Current Pipeline

### Stage 1: Prepare review TeX

Input:
- raw latexdiff `.tex` generated by the `latexdiff` skill

Output:
- copied review `.tex`

Responsibilities:
- copy the raw diff file
- inject the popup macro block into the preamble
- append an empty popup macro after each diff chunk
- skip structural diff chunks such as `\item`, `\begin{...}`, and `\end{...}`
  when direct popup insertion would corrupt the PDF page tree
- assign stable IDs such as `R014-D` and `R014-A`
- print a summary of inserted IDs and write a JSON skeleton that can be filled later

Illustrative shape:

```tex
\DIFdelbegin \DIFdel{in~\cite{...}}\DIFdelend
\DiffPopup{R014-D}{__COMMENT_R014_D__}
\DIFaddbegin \DIFadd{of \citet{...} the }\DIFaddend
\DiffPopup{R014-A}{__COMMENT_R014_A__}
```

### Stage 2: Substantive review pass

Input to Codex:
- a source diff for the same change set, normally `git diff`
- surrounding source or rendered context as needed by the selected review criteria

Output from Codex:
- change-level findings in the style of `paper-review` `review-only`

Constraint:
- keep findings at the level of semantic changes rather than popup targets
- do not force the substantive review pass to reason in terms of add/delete blocks
- one diff hunk may produce multiple findings if it contains separable changes
- one finding may cover multiple changed lines when they form one semantic revision

Illustrative shape:

```md
1. Citation integrated unnaturally into sentence body
   - Issue Location: `section2.tex` line 61
   - Severity: moderate
   - Issue Reason: The parenthetical citation sits awkwardly inside the sentence and makes the prose read mechanically.
   - Revision:
     - Content: none
     - Nuance Shift: none
```

The popup workflow may keep these findings transient. They are an intermediate review result, not automatically a separate deliverable.

### Stage 3: Render popup comments

Input to Codex:
- the prepared review `.tex` as a whole
- the substantive findings from Stage 2

Output from Codex:
- a separate JSON file keyed by ID

Constraint:
- Codex writes only this comment file.
- Codex does not directly rewrite the raw diff `.tex`.
- Codex does not directly rewrite the prepared review `.tex`.
- map change-level findings onto popup IDs only at this stage
- let one finding feed multiple popup IDs when the same semantic change produced both delete and add targets
- if Stage 2 produced no finding for a reviewed change, synthesize a short popup comment directly from the diff target and nearby context instead of leaving the target blank

Illustrative shape:

```json
{
  "R014-D": "旧表現は引用の掛かり先が不自然で、構文上の収まりが悪い。",
  "R014-A": "新表現では著者名を前に出し、文の主部に自然に統合している。"
}
```

Current semantic policy:
- use the same substantive review criteria as `paper-review`; popup review changes the diff-oriented output format, not the underlying judgment standard
- perform the substantive review first in the style of `paper-review` `review-only`
- use `git diff` as the default substantive review surface, unless another diff representation is clearly better for the task
- load only the relevant parts of `../paper-review/references/criteria.md` for the current diff
- keep substantive findings at semantic-change granularity instead of popup-target granularity
- delete comment: describe the role of the old wording and evaluate whether deleting it is appropriate
- add comment: describe the effect of the new wording and evaluate whether adding it is appropriate
- every comment should include a meaning-change label so readers can quickly see whether the diff preserves meaning
- recommended meaning-change labels: `[意味維持]`, `[意味変化あり]`, `[意味変化要確認]`
- prefer explicit review labels such as `[OK]`, `[要確認]`, and `[懸念]` when they help distinguish endorsement, uncertainty, and criticism
- do not assume that every delete is good and every add is justified
- language: follow the user or project preference
- every reviewed diff target must receive some comment text
- when no substantive finding exists, default to a concise `[OK]` fallback comment unless the popup-stage reading itself reveals a concern
- the filled popup-review PDF is the default review record, so do not create an extra Markdown note by default
- intermediate `paper-review` style findings may remain transient unless the user explicitly asks to save them
- if a diff looks inappropriate, still write its per-target comment first and summarize the issue later in a separate document
- create that separate document only when the user explicitly asks for one or when the problematic diff summary is materially useful beyond the popup PDF itself

### Stage 4: Fill comments

Input:
- prepared review `.tex`
- comment file

Output:
- filled review `.tex`

Mechanism:
- string replacement keyed by the stable ID placeholder
- the initial implementation uses a JSON object mapping `ID -> comment`
- before replacement, each comment is encoded as UTF-16BE hex for the PDF annotation string
- the write back into the review `.tex` is performed by the fill script, not by Codex directly

## Diff Targets

Treat all of the following as review targets:

- `\DIFdelbegin ... \DIFdelend`
- `\DIFaddbegin ... \DIFaddend`
- `\DIFdelbeginFL ... \DIFdelendFL`
- `\DIFaddbeginFL ... \DIFaddendFL`

Pairing between delete and add may be used for interpretation, but comments are still attached separately.
The substantive review that feeds those comments should still be done at the level of semantic changes rather than raw delete/add pairs.

Structural exception:
- structural control diffs such as `\item`, `\begin{...}`, `\end{...}`, and
  `\bibitem` may be skipped by the prepare stage, because placing `\pdfannot`
  directly after them can corrupt `/Annots` arrays in the generated PDF.

## Stable Choices

- target viewer: Adobe Acrobat / Acrobat Reader
- popup default: `Open=false`
- primary key: sequential IDs, not line numbers
- popup contents: store comment text as UTF-16BE hex in `/Contents`
- Codex write scope: comment file only
- review criteria source: relevant parts of `paper-review/references/criteria.md`
- substantive review pass: `paper-review` `review-only` style findings, normally over `git diff`
- review completeness: do not leave reviewed targets with empty comments
- default deliverable: the popup-reviewed PDF itself
- issue handling: finish the full comment file first, then create a separate summary document for problematic diffs only when explicitly requested or otherwise clearly needed

## Open Questions
